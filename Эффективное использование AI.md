# Эффективное использование AI: инструкция пользователя

## О чём эта инструкция

Эта инструкция о том, как использовать AI, ИИ, ChatGPT, Claude, Gemini, Perplexity, NotebookLM, Codex, Claude Code, локальные LLM, агенты, скиллы, проекты, память и похожие инструменты как рабочую систему, а не как кнопку "сделай красиво".

Главный вопрос:

```text
Как поставить AI такую задачу, чтобы он помог получить проверяемый, полезный и применимый результат,
а не просто сгенерировал гладкий текст?
```

В эту тему входят:

- постановка задачи для AI;
- prompt engineering / промптинг;
- context engineering / управление контекстом;
- работа с памятью ChatGPT;
- Custom Instructions, Projects, Gems и похожие настройки;
- работа с файлами, документами, источниками и исследованиями;
- source-grounded AI / ответы, заземлённые в источниках;
- борьба с галлюцинациями;
- отделение фактов, гипотез и интерпретаций;
- AI как критик, редактор, тьютор, тренажёр, аналитик, экспертная панель;
- AI для текстов, кода, обучения, решений, планирования, коммуникации и рутины;
- AI-агенты, Codex, Claude Code, Skills, MCP, плагины, хуки;
- NotebookLM, RAG, Wiki LLM и Obsidian-базы;
- "фишки" и рабочие приёмы, которые повышают качество результата;
- ограничения, приватность и безопасность.

Короткая формула:

```text
хороший AI-результат = ясная цель + релевантный контекст + критерии качества + проверка + итерация.
```

## Главная мысль

AI полезнее всего не как оракул, а как усилитель мышления и исполнитель в понятной системе.

Плохой режим:

```text
мутная задача -> AI угадывает -> гладкий ответ -> кажется, что получилось
```

Сильный режим:

```text
цель -> контекст -> формат -> критерии -> черновик -> критика -> проверка -> исправление -> сохранённый процесс
```

AI не знает вашу ситуацию, если вы её не дали. Он может уверенно ошибаться, сглаживать неопределённость, додумывать пробелы и подстраиваться под ожидания пользователя. Поэтому эффективное использование AI начинается не с "волшебного промпта", а с нормальной постановки задачи.

## Базовый словарь

| Термин | Что означает |
|---|---|
| AI / ИИ | Общее название систем искусственного интеллекта. В этой инструкции чаще всего речь о генеративном AI. |
| GenAI | Генеративный ИИ: создаёт текст, код, изображения, аудио, видео или структуру данных. |
| LLM | Large Language Model, большая языковая модель: GPT, Claude, Gemini и похожие модели. |
| Модель | "Мозги": система, которая получает контекст и генерирует ответ. |
| Агент | Обвязка вокруг модели: может работать с файлами, инструментами, командами, памятью и несколькими шагами. |
| Harness | Среда, в которой работает агент: интерфейс, правила, память, инструменты, плагины, циклы выполнения. |
| Промпт | Разовый запрос или инструкция к модели. |
| Скилл | Повторяемая инструкция для типовой задачи. Нужен, когда один и тот же процесс используется много раз. |
| Плагин | Пакет, который может включать скиллы, инструменты, коннекторы и дополнительные возможности. |
| MCP | Model Context Protocol: способ подключать инструменты и сервисы к агенту. |
| Hook / хук | Техническое действие по событию. Надёжнее текстовой просьбы там, где правило критично. |
| Контекстное окно | Объём информации, который модель удерживает в текущей сессии. |
| Context drift | Дрейф контекста: качество ответа падает из-за длинного, смешанного или замусоренного чата. |
| Source-grounded answer | Ответ, опирающийся на конкретные предоставленные источники. |
| Галлюцинация | Уверенно выданный, но недостоверный или выдуманный факт. |
| Automation bias | Склонность чрезмерно доверять ответу автоматической системы. |
| AI-feedback | Обратная связь от AI: проверка черновика, рассуждения, кода, ответа или плана. |
| Metacognitive laziness | Риск, когда AI снимает с человека необходимость самому планировать, проверять и оценивать понимание. |
| Performance-learning gap | Разрыв между текущим выполнением задачи с AI и реальным устойчивым обучением. |
| Outcome-first prompt | Промпт, который сначала описывает желаемый результат, критерии качества и границы. |
| Stopping condition | Условие остановки: когда результат уже достаточно хорош и модель должна не расширять задачу. |

## Карта эффективного использования AI

| Задача | Хорошая роль AI | Главный риск | Правильный контроль |
|---|---|---|---|
| Разобрать хаос | Структурировщик | Потеря деталей | Проверить, не исчезли ли ограничения |
| Написать текст | Черновик + редактор | Шаблонный AI-слог | Сверить с целью, фактами и стилем |
| Принять решение | Аналитик + критик | Ложная уверенность | Сравнить варианты, риски, неизвестные |
| Учиться | Тьютор, экзаменатор, feedback | Готовый ответ вместо мышления | Своя попытка до AI, повтор без AI |
| Работать с источниками | Аудитор, извлекатель, суммаризатор | Галлюцинации и неверные ссылки | Цитаты, доказательства, статус вывода |
| Кодить | Наставник, reviewer, генератор тестов | Нерабочий код | Запуск, тесты, линтеры, реальные ошибки |
| Автоматизировать | Агент с инструментами | Повреждение файлов, утечки, неверные действия | Копии, Git, разрешения, план до изменений |
| Работать с памятью | Персональный контекст | Устаревшие или неверные сведения о вас | Периодический аудит памяти |
| Создавать систему | Скиллы, шаблоны, wiki | Беспорядок из промптов | Версионировать удачные процессы |

## Базовый протокол работы с AI

Этот протокол подходит почти для любой задачи: текст, код, обучение, документы, исследования, планирование, юридический черновик, деловое письмо, обработка заметок, подготовка к разговору, Obsidian, агентная работа.

### Шаг 1. Назови результат

Не начинай с общего:

```text
Помоги с этим.
```

Лучше:

```text
Мне нужен [конкретный результат]:
- таблица сравнения;
- план на 7 дней;
- черновик письма;
- список рисков;
- code review;
- конспект;
- набор вопросов;
- инструкция;
- проверка источников;
- список следующих действий.
```

AI должен понимать, какой артефакт должен получиться.

### Шаг 2. Дай контекст

Минимальный контекст:

```text
Контекст:
- что происходит;
- зачем нужен результат;
- кто будет использовать результат;
- что уже есть;
- что уже пробовал;
- какие ограничения важны;
- чего нельзя делать;
- какой уровень глубины нужен.
```

Если контекста мало, AI будет угадывать. Иногда угадывание выглядит убедительно, но не попадает в реальную задачу.

### Шаг 3. Укажи формат

Формат резко повышает качество.

Примеры форматов:

- таблица;
- чек-лист;
- матрица решений;
- пошаговая инструкция;
- список рисков;
- "проблема -> причина -> исправление";
- "вывод -> почему -> что делать дальше";
- "факты -> гипотезы -> неизвестные зоны -> следующий шаг";
- Markdown-файл;
- JSON;
- код с тестами;
- план работ.

Плохой запрос:

```text
Расскажи про варианты.
```

Лучше:

```text
Сравни варианты A и B в таблице.
Столбцы: критерий, вариант A, вариант B, что лучше, почему, риск.
Без текста вне таблицы.
```

### Шаг 4. Задай критерии качества

AI часто не знает, что именно вы считаете хорошим результатом.

Критерии могут быть такими:

- применимо на практике;
- без общих фраз;
- не добавляет новых фактов;
- отделяет факты от предположений;
- указывает риски;
- сохраняет спокойный стиль;
- даёт следующий шаг;
- даёт проверочные вопросы;
- не расширяет задачу;
- укладывается в объём;
- подходит для конкретной аудитории;
- можно использовать без исходника.

### Шаг 5. Попроси вопросы до ответа

Для сложных задач полезно не просить AI сразу выполнять работу.

Шаблон:

```text
Сначала не выполняй задачу.
Проверь, достаточно ли данных.
Если данных не хватает, задай до 5 уточняющих вопросов.
Если данных достаточно, кратко повтори мою задачу и предложи план.
```

Это защищает от ситуации, когда модель уверенно решает не ту задачу.

### Шаг 6. Сначала план, потом исполнение

Для важных задач:

```text
Сначала предложи план работы.
Не пиши финальный результат, пока я не подтвержу план.
```

Это особенно важно для агентов, которые могут менять файлы, запускать команды, редактировать документы или работать с локальной папкой.

### Шаг 7. Проверь ответ

После результата проверь:

- решена ли исходная задача;
- не добавлены ли факты без основания;
- не смешаны ли факты и гипотезы;
- нет ли слишком сильных выводов;
- соответствует ли формат;
- можно ли это применить;
- что нужно проверить внешне;
- где AI мог ошибиться;
- нужен ли второй проход.

### Шаг 8. Сделай критику

Не принимай первый ответ как финальный.

Рабочая схема:

```text
ответ -> критика -> исправленная версия
```

Промпт:

```text
Проверь свой предыдущий ответ критически.
Найди:
1. слабые места;
2. скрытые допущения;
3. неподтверждённые факты;
4. места, где вывод слишком сильный;
5. что нужно проверить.

После этого дай исправленную версию.
```

### Шаг 9. Сохрани удачный процесс

Если результат получился хорошим, не оставляй его случайностью.

Сохрани:

- промпт;
- контекст;
- формат ответа;
- критерии качества;
- типичные ошибки;
- итоговый шаблон;
- если задача повторяется, оформи как скилл или инструкцию.

## Универсальный промпт

```text
Действуй как [роль].

Задача:
[что нужно сделать]

Контекст:
[факты, ситуация, цель, что уже есть, что пробовал]

Аудитория:
[для кого результат]

Формат результата:
[таблица / чек-лист / план / инструкция / код / Markdown / JSON]

Критерии качества:
- [критерий 1]
- [критерий 2]
- [критерий 3]

Ограничения:
- не добавляй факты, которых нет в исходных данных;
- отделяй факты от предположений;
- если данных недостаточно, сначала укажи, чего не хватает;
- не расширяй задачу без необходимости.

Условие остановки:
[когда результат считается достаточным]

Сначала проверь, достаточно ли данных. Если нет - задай до 5 вопросов.
```

## Короткая версия универсального промпта

```text
Контекст: [ситуация].
Цель: [результат].
Ограничения: [что важно / чего нельзя].
Формат: [как выдать ответ].
Критерии: [что будет хорошим результатом].
Если данных недостаточно - не выдумывай, задай вопросы.
```

## Outcome-first: промпт от результата

Для современных моделей часто лучше не микроменеджерить весь процесс, а ясно описать результат.

Плохой вариант:

```text
Сначала сделай A, потом B, потом C, потом D, потом обязательно E...
```

Лучше:

```text
Нужен результат: [X].

Хороший результат:
- решает [задачу];
- учитывает [ограничения];
- опирается на [источники / данные];
- выдаётся в формате [формат];
- не содержит [запрет].

Остановись, когда:
- [критерий завершения].
```

Процесс стоит прописывать подробно только там, где порядок действий действительно критичен: безопасность, формат, API-схемы, юридическое извлечение фактов, автоматизация файлов, кодовые миграции.

## Настройка AI до запроса

Если вы часто используете AI, часть правил лучше вынести в постоянные настройки:

- Custom Instructions в ChatGPT;
- Projects в ChatGPT и Claude;
- Gems в Gemini;
- системные инструкции в агентной среде;
- `AGENTS.md`, `CLAUDE.md`, `SKILL.md` в локальном проекте;
- память ChatGPT, если она доступна и уместна.

Базовая инструкция:

```text
Отвечай спокойно, структурно и практически.
Начинай с главного вывода.
Если вопрос простой - отвечай кратко.
Если тема сложная - сначала дай краткий вывод, затем структуру.
Отделяй факты от предположений.
Не соглашайся автоматически.
Если данных недостаточно - задай уточняющий вопрос или явно обозначь ограничение.
Не добавляй факты, которых нет в исходных данных.
В важных темах указывай, что нужно проверить отдельно.
```

Для краткости лучше задавать измеримые правила:

```text
Простые вопросы - 1-2 предложения.
Средние вопросы - 3-5 предложений или короткий список.
Сложные темы - краткий вывод, затем структура.
Если нужен выбор - дай рекомендацию, критерии и главный риск.
```

## Контекст-инжиниринг

Контекст-инжиниринг - это не "засунуть в модель всё подряд". Это умение отобрать, сжать и расположить информацию так, чтобы модель работала с нужными данными.

### Что давать в контекст

Для любой серьёзной задачи полезно дать:

- цель;
- исходные данные;
- аудиторию;
- ограничения;
- важные факты;
- неизвестные зоны;
- что уже пробовали;
- что нельзя менять;
- образец хорошего результата;
- образец плохого результата, если стиль важен.

### Что не давать в контекст

Не стоит без необходимости вставлять:

- огромные переписки целиком;
- старые противоречивые версии;
- нерелевантные эмоции;
- длинные логи без выделения ошибки;
- документы без указания, что в них искать;
- десятки требований без приоритета;
- старые инструкции, которые конфликтуют с новой задачей.

### Сырой контекст через голос

Если руками писать долго, можно наговорить сырой контекст.

Рабочий цикл:

1. Наговорить ситуацию голосом.
2. Попросить AI не решать задачу, а структурировать контекст.
3. Проверить, не потерялись ли важные детали.
4. Сформировать финальный запрос.
5. Получить ответ.
6. Проверить факты, применимость и следующий шаг.

Шаблон:

```text
Я сейчас дам сырой контекст.
Не решай задачу сразу.
Сначала выдели:
1. цель;
2. ограничения;
3. факты;
4. неизвестные зоны;
5. недостающие данные;
6. возможный формат финального запроса.
```

### Сжатие контекста

Для длинных материалов:

```text
Сожми следующий материал для дальнейшей работы.

Сохрани:
1. факты;
2. даты;
3. имена и роли;
4. числовые данные;
5. решения;
6. противоречия;
7. неизвестные зоны;
8. ограничения;
9. цитаты, если они критичны.

Удали:
- повторы;
- фоновые рассуждения;
- нерелевантные фрагменты;
- эмоциональные отступления.

Формат:
## Краткая суть
## Факты
## Ограничения
## Противоречия
## Неизвестные зоны
## Что важно для следующего запроса
```

## Одна задача - один чат

Смешивание тем ухудшает качество.

Правило:

```text
одна задача -> один чат -> один контекст
```

Новый чат нужен, если:

- тема изменилась;
- старый чат перегружен;
- в диалоге накопилось много лишнего текста;
- модель начала тащить старые предположения;
- нужна чистая фильтрация большого материала;
- нужно отделить черновой брейншторм от финального результата.

Для длинных задач:

```text
Сделай подробное summary текущего контекста:
1. цель;
2. что уже решено;
3. важные факты;
4. принятые решения;
5. открытые вопросы;
6. текущий план;
7. правила, которые важно сохранить.
```

После этого можно продолжить в новом чате.

## Основные режимы использования AI

### 1. AI как структурировщик

Использовать, когда есть хаос:

- мысли кучей;
- длинная переписка;
- заметки;
- транскрипт;
- документ;
- список задач;
- конфликтная ситуация;
- техническая ошибка;
- учебная тема.

Шаблон:

```text
Разбери материал.

Формат:
1. Главная суть.
2. Факты.
3. Выводы.
4. Противоречия.
5. Риски.
6. Неизвестные зоны.
7. Следующие действия.

Не добавляй факты, которых нет в материале.
```

### 2. AI как объяснитель

Использовать для понимания темы.

Шаблон:

```text
Объясни тему: [тема].

Мой уровень: [уровень].
Цель: понять так, чтобы применить на практике.

Формат:
1. Простая ментальная модель.
2. Какую проблему решает.
3. Мини-пример.
4. Типичные ошибки.
5. Проверочный вопрос.
6. Упражнение.
```

Ограничение: объяснение не равно навыку. После объяснения нужна самостоятельная попытка.

### 3. AI как тьютор

Использовать, когда нужно учиться, а не просто получить ответ.

Шаблон:

```text
Будь тьютором.
Не давай готовый ответ сразу.
Сначала спроси, как я понимаю тему.
Потом задай 3-5 вопросов.
После моего ответа:
1. покажи, что верно;
2. найди ошибку;
3. дай подсказку;
4. предложи похожую задачу;
5. проверь повторную попытку.
```

Сильный цикл:

```text
моя попытка -> feedback -> исправление -> похожая задача -> повтор без AI
```

### 4. AI как экзаменатор

Использовать для проверки понимания.

```text
Проверь моё понимание темы [тема].

Задавай вопросы по одному.
Начни с простых, затем усложняй.
После каждого ответа:
- оцени точность;
- укажи пробел;
- дай короткую подсказку;
- не переходи дальше, пока я не исправлю ошибку.
```

Это полезнее пассивного чтения ответа AI.

### 5. AI как feedback-система

Использовать после черновика, попытки, решения, кода, текста.

```text
Вот моя попытка:
[вставить]

Не переписывай всё сразу.
Сначала:
1. найди одну главную проблему;
2. объясни, почему она важна;
3. предложи минимальное исправление;
4. дай критерий проверки;
5. предложи похожую задачу для повторной попытки.
```

Главное ограничение: feedback должен идти после собственной попытки, а не вместо неё.

### 6. AI как критик

Использовать для планов, решений, текстов, стратегий, убеждений.

```text
Проверь мой план как критик.
Не соглашайся автоматически.

Найди:
1. слабые места;
2. скрытые допущения;
3. противоречия;
4. риски;
5. ситуации, где план не сработает;
6. что нужно проверить фактами.

Оцени силу каждого замечания.
Не создавай искусственный баланс, если аргумент слабый.
```

### 7. AI как спарринг-партнёр

Использовать, когда нужно не просто раскритиковать, а честно увидеть разные стороны.

```text
Разбери мою позицию в 4 шага:

1. Steelman: самая сильная версия моей позиции.
2. Devil's advocate: самая сильная критика.
3. Скрытые допущения.
4. Синтез: что сохранить, уточнить или пересмотреть.

Отделяй факты от предположений.
```

### 8. AI как панель экспертов

Использовать для сложных задач, где важны разные перспективы.

```text
Проанализируй задачу методом нескольких экспертных перспектив.

Задача:
[описание]

Сделай:
1. Определи 3-5 экспертных ролей, подходящих для задачи.
2. Для каждой роли покажи:
   - что она считает главным;
   - какие риски видит;
   - какие решения предлагает;
   - какие ограничения учитывает.
3. Сравни позиции:
   - где согласны;
   - где расходятся;
   - какие противоречия важны.
4. Собери общий практический план.

Не выдумывай недостающие данные.
```

Оптимально 3-5 ролей. Больше часто размазывает ответ.

### 9. AI как редактор

Использовать для текстов, писем, заявлений, постов, инструкций, резюме.

```text
Отредактируй текст.

Цель:
[что должен сделать текст]

Стиль:
[спокойный / деловой / юридически осторожный / живой / краткий]

Ограничения:
- не добавляй новых фактов;
- не усиливай утверждения;
- сохрани смысл;
- убери канцелярит и AI-слог;
- отметь места, где текст слишком общий или недоказанный.

Формат:
1. Исправленная версия.
2. Главные правки.
3. Что осталось спорным.
```

### 10. AI как автор черновика

AI может писать первый черновик, но финальная ответственность остаётся на человеке.

Хороший режим:

```text
моя мысль -> AI-черновик -> моя редактура -> AI-критика -> финальная версия
```

Плохой режим:

```text
AI написал -> я скопировал -> готово
```

### 11. AI как исследователь

Использовать, когда нужно собрать источники, сравнить позиции, найти противоречия.

Важно: если тема актуальная или высокорисковая, нужен доступ к свежим источникам и проверка.

Шаблон:

```text
Собери обзор по теме [тема].

Требования:
1. Отдели подтверждённые факты от интерпретаций.
2. Укажи даты источников.
3. Покажи, где источники противоречат друг другу.
4. Укажи слабые места доказательств.
5. Не делай сильных выводов без достаточных данных.
6. В конце дай список того, что нужно проверить отдельно.
```

### 12. AI как аудитор источников

Перед содержательной работой с большой базой источников запусти три промпта.

```text
1. Проанализируй источники и составь таблицу:
источник, год, тип, краткая суть, насколько можно доверять, ограничения.
```

```text
2. Найди реальные противоречия между источниками.
Укажи, какие именно источники расходятся и в чём.
Не придумывай расхождения ради полноты.
```

```text
3. Какие важные взгляды, подходы или группы источников отсутствуют?
Где стоит поискать недостающую перспективу?
```

### 13. AI как извлекатель фактов

Для договоров, исследований, таблиц, новостей, документов.

Главное правило:

```text
лучше пустое поле, чем выдуманный ответ.
```

Шаблон:

```text
Работай только с предоставленным источником.
Не используй внешние знания, память модели и догадки.

Если данных недостаточно, значение неоднозначно или вывод требует предположения,
пиши "Недостаточно данных".

Для каждого утверждения укажи:
1. утверждение;
2. статус:
   - прямо указано;
   - переформулировано;
   - следует из явно указанных фактов;
   - гипотеза;
   - недостаточно данных;
3. основание: короткая цитата или указание фрагмента;
4. уровень уверенности;
5. что осталось неизвестным.
```

### 14. AI как помощник по программированию

AI полезен для кода, если код потом проверяется.

Хороший цикл:

```text
гипотеза -> код -> запуск -> ошибка -> AI-feedback -> исправление -> тест
```

Шаблон для ошибки:

```text
Действуй как backend-разработчик и инженер-наставник.

Контекст:
- язык:
- стек:
- цель:
- что ожидал:
- что получил:
- что уже пробовал:

Код:
[код]

Ошибка:
[полный текст ошибки]

Сделай:
1. объясни причину;
2. найди минимальное исправление;
3. объясни, почему оно работает;
4. укажи, как проверить;
5. предложи один тест;
6. назови типичную ошибочную ментальную модель.
```

Не проси только:

```text
исправь код
```

Проси:

```text
объясни ошибку, исправь минимально, покажи проверку.
```

### 15. AI как code reviewer

```text
Проведи code review.

Контекст:
[для чего код]

Проверь:
1. корректность;
2. читаемость;
3. обработку ошибок;
4. тестируемость;
5. edge cases;
6. production-риски;
7. наблюдаемость;
8. поддерживаемость.

Формат:
- критичные проблемы;
- важные улучшения;
- мелкие замечания;
- что проверить тестами;
- улучшенная версия, если нужна.
```

Для более глубокого review можно использовать экспертные роли:

- senior backend-разработчик;
- QA-инженер;
- архитектор;
- SRE / DevOps;
- security reviewer.

### 16. AI как помощник по решениям

```text
Мне нужно выбрать между:
A: ...
B: ...
C: ...

Сравни по критериям:
- польза;
- риск;
- цена ошибки;
- обратимость;
- скрытые издержки;
- влияние через 1 месяц;
- влияние через 1 год;
- что можно проверить малым тестом.

В конце дай:
1. когда лучше A;
2. когда лучше B;
3. когда лучше C;
4. что нужно узнать перед решением;
5. минимальный следующий шаг.
```

AI не должен принимать решение вместо вас. Он должен сделать поле решения яснее.

### 17. AI как помощник по коммуникации

Использовать для:

- сложных разговоров;
- деловых писем;
- претензий;
- переговоров;
- собеседований;
- конфликтов;
- подготовки тезисов.

Шаблон:

```text
Помоги подготовить коммуникацию.

Контекст:
[ситуация]

Цель:
[что нужно получить]

Ограничения:
- не усиливать конфликт;
- не признавать лишнего;
- не добавлять факты;
- сохранить спокойный тон;
- оставить возможность для переговоров.

Сделай:
1. ключевые тезисы;
2. возможные риски формулировок;
3. мягкую версию;
4. нейтральную версию;
5. более жёсткую, но корректную версию.
```

### 18. AI как помощник по самоанализу

Использовать можно, но осторожно.

Подходит:

- структурировать переживание;
- отделить факты от интерпретаций;
- подготовиться к разговору со специалистом;
- составить список вопросов;
- найти возможные когнитивные искажения;
- сформулировать маленький следующий шаг.

Не подходит:

- заменять психотерапию при тяжёлых состояниях;
- диагностировать себя;
- подбирать лечение;
- решать кризисные ситуации без помощи людей;
- слепо принимать интерпретацию AI о ваших мотивах.

Шаблон:

```text
Помоги структурировать ситуацию.
Не ставь диагнозов.
Не утверждай, что точно знаешь мои мотивы.

Раздели:
1. факты;
2. мои интерпретации;
3. эмоции;
4. возможные альтернативные объяснения;
5. что стоит обсудить со специалистом;
6. один маленький безопасный следующий шаг.
```

### 19. AI как помощник по памяти

AI может хранить или восстанавливать контекст, но его память нужно проверять.

Полезные запросы:

```text
Сделай мета-карту того, что ты знаешь обо мне.
Отдельно вынеси:
1. устойчивые факты;
2. предположения;
3. что могло устареть;
4. какие заблуждения обо мне у тебя могут быть;
5. какие вопросы стоит уточнить.
```

```text
Опираясь на память, задай мне до 3 вопросов,
которые помогут тебе лучше понимать мои цели, ограничения и стиль работы.
```

Важно: память модели не является точной биографией. Она может быть устаревшей, неполной или ошибочно обобщённой.

## Как выбирать инструмент

| Инструмент | Где особенно полезен | Осторожность |
|---|---|---|
| ChatGPT | постановка задач, объяснения, тексты, анализ, код, повседневные задачи | проверять факты и свежесть |
| Claude | длинные документы, тексты, аналитика, проекты, coding workflows | не путать гладкость с точностью |
| Gemini / Deep Research | поиск, источники, большие корпуса, связка с Google-средой | аудит источников обязателен |
| Perplexity | быстрый поиск с источниками | проверять качество источников |
| NotebookLM | работа по загруженным источникам, обучение, аудиообзоры, critique/debate | источник всё равно нужно проверять |
| Codex / Claude Code | работа с файлами, кодом, проектами, агентные задачи | копии, Git, разрешения, план до изменений |
| Локальные LLM | приватность, локальные сценарии, эксперименты | качество может быть ниже, нужна настройка |
| Локальные scripts | повторяемая обработка Markdown, таблиц, файлов | требуют проверки и поддержки |

Выбор лучше делать от задачи, а не от моды на модель.

## Работа с источниками, RAG, NotebookLM и Wiki

### Когда использовать

Использовать, если нужно:

- разобрать большой набор документов;
- анализировать исследования;
- готовить конспект по источникам;
- искать противоречия;
- строить базу знаний;
- работать с PDF, заметками, регламентами, транскриптами;
- готовиться к встрече или экзамену по корпусу материалов.

### Главное правило

AI по источникам должен отвечать на основе источников, а не "из головы".

Промпт:

```text
Отвечай только на основе загруженных источников.
Если в источниках нет ответа, так и скажи.
Для каждого важного утверждения укажи источник или фрагмент.
Отделяй:
- прямо указано;
- логически следует;
- гипотеза;
- недостаточно данных.
```

### NotebookLM: полезные режимы

NotebookLM полезен не только как "сделать аудиообзор".

Практические режимы:

- Learning Guide: AI сначала спрашивает, что вы знаете, и превращается в экзаменатора;
- Brief: короткая выжимка перед встречей;
- Critique: поиск логических дыр и неудобных вопросов;
- Debate: столкновение позиций "за" и "против";
- Interactive Mode: живое уточнение в аудиообзоре, если функция доступна;
- Revise: точечная правка слайда или части результата без полной перегенерации;
- Google Drive integration: быстрый доступ к источникам, если вы работаете в этой экосистеме.

Ограничение: не все функции доступны везде и на всех языках. Проверяйте интерфейс и актуальные возможности.

### Хак со сжатием источников

Если есть лимит источников, можно группировать документы:

1. Загрузить 5-10 источников.
2. Попросить подробное структурированное резюме.
3. Сохранить резюме как отдельный источник.
4. Удалить исходные файлы из блока, если это допустимо для задачи.
5. Повторить с другими группами.

Осторожность: сжатие может потерять детали. Для юридических, научных, медицинских и финансовых задач важные первоисточники лучше сохранять отдельно.

### Wiki LLM вместо хаотичного RAG

Подход Wiki LLM:

```text
raw -> wiki -> index -> log -> вопросы к базе
```

Где:

- `raw` - исходные файлы;
- `wiki` - обработанные смысловые страницы;
- `index.md` - карта базы;
- `log.md` - журнал изменений;
- `AGENTS.md` / `CLAUDE.md` - правила обработки.

Пользователь отвечает за:

- выбор источников;
- контроль результата;
- проверку ошибок;
- уточнение схемы;
- резервные копии;
- приватность.

Модель помогает:

- обработать новые файлы;
- выделить концепты;
- связать страницы;
- обновить индекс;
- ответить на вопросы по локальной базе.

## Работа с текстами

### AI-черновик

Для черновика:

```text
Напиши черновик текста.

Цель:
[что должен сделать текст]

Аудитория:
[кто читает]

Стиль:
[тон, уровень формальности]

Факты, которые можно использовать:
[список]

Ограничения:
- не добавляй новых фактов;
- не делай рекламный стиль;
- не усиливай утверждения;
- если данных не хватает, оставь место или задай вопрос.
```

### Редактура против AI-слога

Искать и убирать:

- "важно отметить";
- "следует подчеркнуть";
- "играет ключевую роль";
- "является важным аспектом";
- гладкие абзацы без конкретики;
- одинаковую структуру предложений;
- синонимическую карусель;
- списки ради списков;
- чрезмерную нейтральность;
- выводы без указания неопределённости.

Промпт:

```text
Отредактируй текст так, чтобы убрать AI-слог.

Исправь:
1. пустые вводные;
2. канцелярит;
3. чрезмерную гладкость;
4. общие фразы без содержания;
5. повторы;
6. искусственную симметрию;
7. выводы без оснований.

Не добавляй новых фактов.
Сохрани смысл и спокойный профессиональный тон.
```

### Авторский стиль

Если нужно, чтобы AI писал ближе к вашему стилю, не пишите просто:

```text
Пиши как я.
```

Лучше:

1. Дать 5-10 хороших текстов.
2. Попросить выделить признаки стиля.
3. Создать документ "Мой авторский стиль".
4. После каждого слабого результата сравнить AI-версию и вашу правку.
5. Обновить инструкцию.

Шаблон:

```text
Проанализируй мои тексты и опиши мой авторский стиль:
1. как я начинаю;
2. как строю предложения;
3. какие слова использую;
4. какой тон;
5. как объясняю сложные вещи;
6. чего избегать;
7. чем мой стиль отличается от типичного AI-текста.

Оформи как инструкцию для будущих текстов.
```

## Работа с обучением

AI в обучении полезен, если усиливает:

- active recall;
- self-explanation;
- feedback;
- error correction;
- transfer;
- spaced repetition;
- метакогницию.

AI в обучении вреден, если заменяет:

- попытку;
- ошибку;
- извлечение из памяти;
- самостоятельное объяснение;
- проверку;
- повтор без подсказки.

### Правило AI after attempt

```text
сначала моя попытка -> потом AI
```

Минимум перед запросом:

- 5 строк своего объяснения;
- черновик решения;
- прогноз;
- вопрос;
- список того, где застрял;
- код до ошибки.

Шаблон:

```text
Я изучаю [тема].
Не давай готовый ответ сразу.

Моя попытка:
[текст]

Сделай:
1. найди главную ошибку;
2. задай наводящий вопрос;
3. дай минимальную подсказку;
4. предложи похожую задачу;
5. проверь мой повторный ответ.
```

### Почему это важно

AI может улучшить текущий результат, но не всегда формирует устойчивое learning. Если после помощи AI вы не можете повторить действие без него, вырос performance, а не навык.

Контрольный вопрос:

```text
После этой AI-сессии я могу сделать похожую задачу без AI?
```

## Работа с агентами

AI-агент отличается от обычного чата тем, что может работать в среде:

- читать файлы;
- создавать файлы;
- запускать команды;
- редактировать проект;
- обращаться к инструментам;
- делать несколько шагов;
- сохранять результат.

### Когда нужен агент

Агент нужен, если задача:

- многошаговая;
- повторяемая;
- связана с файлами;
- требует проверки результата;
- требует запуска команд;
- требует изменения проекта;
- должна сохранить результат в нужном месте.

Для простого вопроса агент не нужен.

### Первый запрос агенту

```text
Я работаю над задачей:
[одна конкретная задача]

Материалы:
- [файл/папка] - что внутри;
- [шаблон] - как должен выглядеть результат.

Ограничения:
- не меняй оригиналы;
- работай только в этой папке;
- не используй личные данные;
- перед изменениями покажи план;
- не запускай опасные действия без подтверждения.

Ожидаемый результат:
[файл / отчёт / таблица / исправленный код]

Сначала проверь, правильно ли ты понял задачу.
Если данных не хватает, задай вопросы.
Затем предложи план.
```

### Безопасность агента

На старте:

- работайте с копиями;
- используйте Git;
- не давайте пароли;
- не давайте личные данные без необходимости;
- не нажимайте "Always Allow" механически;
- читайте разрешения;
- ограничивайте папку;
- просите план до изменений;
- останавливайте агента, если он пошёл не туда.

Фраза остановки:

```text
Стоп. Ты пошёл не туда.
Не продолжай выполнение.
Сначала объясни, что ты собирался сделать и зачем.
Предложи новый план с учётом моего замечания.
```

### Если агент застрял

Спросите:

```text
Что тебе мешает выполнить задачу?
Каких данных не хватает?
Какого инструмента или доступа не хватает?
Какие есть 2-3 альтернативных пути?
Какой самый безопасный следующий шаг?
```

## Скиллы и повторяемые процессы

Если вы регулярно копируете один и тот же промпт, это кандидат в скилл.

Скилл нужен для:

- еженедельных отчётов;
- обработки источников;
- подготовки Markdown-инструкций;
- проверки текстов;
- code review;
- анализа документов;
- создания конспектов;
- исследования конкурентов;
- подготовки презентаций;
- обработки файлов в Obsidian.

### Как создать скилл из успешной работы

После хорошего результата:

```text
Мы сейчас успешно выполнили задачу.
Собери из этого повторяемый скилл.

Опиши:
1. когда его использовать;
2. какие входные данные нужны;
3. порядок работы;
4. критерии качества;
5. типичные ошибки;
6. ограничения безопасности;
7. формат результата;
8. примеры хорошего результата.
```

### Проверка сторонних скиллов

Перед использованием чужого скилла:

- прочитать самому;
- попросить AI найти подозрительные инструкции;
- проверить, не просит ли он секреты, ключи, личные данные;
- не устанавливать непонятные скиллы в важную среду;
- тестировать на копиях.

Шаблон проверки:

```text
Проверь этот skill.md на риски.

Ищи:
1. попытки получить секреты;
2. передачу данных наружу;
3. опасные команды;
4. скрытое изменение файлов;
5. конфликтующие инструкции;
6. чрезмерно широкие разрешения;
7. места, где нужен ручной контроль.
```

## Хуки, ограничения и технические барьеры

Текстовая инструкция полезна, но не всегда достаточна.

Если правило критично, лучше закреплять его технически:

- права доступа;
- Git;
- резервные копии;
- hooks;
- pre-commit checks;
- запреты на опасные команды;
- ограниченная рабочая папка;
- ручные подтверждения;
- тесты перед публикацией.

Примеры критичных правил:

- не удалять файлы без подтверждения;
- не пушить без тестов;
- не отправлять секреты наружу;
- не изменять оригиналы;
- не выходить за рабочую папку;
- не трогать медицинские, юридические или финансовые данные без контроля.

## Большой блок фишек

### Фишка 1. Контракт перед задачей

```text
Я дам сырой контекст.
Твоя задача - сначала выделить цель, ограничения, недостающие данные и критерии качества.
Не выполняй финальную задачу, пока не сформулируешь рабочее ТЗ.
```

### Фишка 2. "Задай вопросы, пока не готов"

```text
Если данных недостаточно, не отвечай.
Задавай уточняющие вопросы, пока не сможешь выполнить задачу с приемлемой уверенностью.
Максимум 5 вопросов за раз.
```

Формулировка "95% уверенности" может быть полезной эвристикой, но не реальной измеримой вероятностью.

### Фишка 3. Структура -> критика -> синтез

```text
Разбери по схеме:
1. структура;
2. критика и ограничения;
3. альтернативы;
4. риски;
5. синтез и следующий шаг.
```

### Фишка 4. "Убеди меня в обратном"

```text
Теперь найди аргументы против своего предыдущего вывода.
Покажи, в каких условиях он не работает.
Оцени силу контраргументов.
Не спорь ради спора.
```

Важно: это не фактчек, а смена перспективы.

### Фишка 5. Несколько экспертов

```text
Разбери задачу через 4 роли:
1. эксперт по предмету;
2. критик рисков;
3. практик-исполнитель;
4. пользователь результата.

Покажи, где они согласны и где спорят.
```

### Фишка 6. Сначала диагностический проход

```text
Пока не решай задачу.
Сначала определи:
1. что это за задача;
2. какие данные важны;
3. какие ограничения есть;
4. какие ошибки возможны;
5. какой формат результата лучше.
```

### Фишка 7. Пустое поле лучше ложного

Для извлечения фактов:

```text
Если в источнике нет данных, пиши "Недостаточно данных".
Неверный ответ хуже пустого.
```

### Фишка 8. Колонка "основание"

В таблицах по источникам добавляй столбцы:

- значение;
- статус;
- основание;
- уверенность;
- что неизвестно.

### Фишка 9. Колонка "почему пусто"

Если поле не заполнено, модель должна указать причину:

- нет данных;
- данные противоречат;
- неоднозначно;
- нужна внешняя информация.

### Фишка 10. Фильтр-проход в новом чате

Если старый чат раздулся:

1. Скопировать важный материал.
2. Открыть новый чат.
3. Попросить извлечь только факты, выводы, риски и действия.
4. Запретить добавлять новые факты.

### Фишка 11. Заточка ответа

```text
Сократи предыдущий ответ на 50%.
Оставь только факты, выводы, действия и риски.
Убери вступления, повторы и общие рассуждения.
Не добавляй новые идеи.
```

### Фишка 12. Жёсткая форма вместо "будь кратким"

Плохо:

```text
Ответь кратко.
```

Лучше:

```text
Ответь в 5 пунктах.
Каждый пункт - одно предложение.
Без вступления и финального резюме.
```

### Фишка 13. "Без текста вне таблицы"

Для сравнения:

```text
Ответь только таблицей.
Без текста до и после таблицы.
```

### Фишка 14. Реверсивный промптинг

Если результат плохой:

```text
Вот результат AI.
Вот результат, который я считаю правильным.

Сравни:
1. чем они отличаются;
2. какой информации не хватало;
3. как обновить промпт;
4. какие правила добавить на будущее.
```

### Фишка 15. "Как бы ты решал эту задачу?"

Когда неясно, с чего начать:

```text
Если бы у тебя стояла эта задача, как бы ты её решал?
Какие вопросы задал бы перед началом?
Какие материалы запросил бы?
Какой порядок работы предложил бы?
```

### Фишка 16. Паспорт проекта

Создать документ с:

- кто вы;
- аудитория;
- продукты / задачи;
- стиль;
- цели;
- ограничения;
- примеры хороших результатов;
- типичные ошибки;
- что AI должен знать перед работой.

Шаблон:

```text
Помоги создать паспорт проекта.
Сначала задай вопросы о моей экспертности, аудитории, продуктах, стиле, целях и ограничениях.
Затем оформи ответы в структурированный документ для новых чатов.
```

### Фишка 17. Паспорт стиля

Дать AI 5-10 своих текстов и попросить создать инструкцию по стилю.

Потом обновлять её через сравнение:

```text
Вот AI-версия.
Вот моя исправленная версия.
Обнови инструкцию по стилю на основе различий.
```

### Фишка 18. Память с аудитом

```text
Что ты знаешь обо мне?
Раздели:
1. подтверждённые факты;
2. предположения;
3. что могло устареть;
4. где ты можешь ошибаться;
5. что стоит уточнить.
```

### Фишка 19. AI как адвокат дьявола

```text
Я планирую [цель].
Назови 3 сильные причины, по которым я могу потерпеть неудачу.
Для каждой:
- почему это риск;
- ранний сигнал;
- как снизить риск.
```

### Фишка 20. Pre-mortem

```text
Представь, что проект провалился через 3 месяца.
Назови 10 реалистичных причин провала.
Раздели их на:
- управляемые;
- частично управляемые;
- внешние.
Для управляемых дай защитные действия.
```

### Фишка 21. Мини-эксперимент на 7 дней

```text
Помоги проверить этот AI-workflow за 7 дней.

Сформулируй:
1. гипотезу;
2. одно изменяемое правило;
3. простые метрики;
4. критерий оставить / изменить / убрать;
5. ежедневный шаблон записи.
```

### Фишка 22. AI как генератор тестов

Для кода, знаний, планов:

```text
Сделай тесты, которые могут опровергнуть мой результат.
Не подтверждай, а ищи случаи, где он ломается.
```

### Фишка 23. AI как генератор критериев

Перед работой:

```text
Сначала предложи критерии качества результата.
После моего подтверждения выполни задачу.
В конце проверь результат по этим критериям.
```

### Фишка 24. AI как "следующее действие"

Когда ответ слишком общий:

```text
Сформулируй один маленький следующий шаг,
который можно выполнить за 10-20 минут.
Без мотивационных фраз.
```

### Фишка 25. AI для жалоб и автоматизации

Если не знаете, что автоматизировать:

```text
Вот список задач, которые меня раздражают:
[список]

Найди:
1. что повторяется;
2. что можно шаблонизировать;
3. что можно автоматизировать;
4. что можно делегировать AI;
5. какой самый маленький прототип сделать.
```

### Фишка 26. AI как переводчик между хаосом и системой

```text
Преврати этот хаотичный список в систему:
1. категории;
2. повторяющиеся паттерны;
3. зависимости;
4. действия;
5. что удалить;
6. что отложить.
```

### Фишка 27. AI для подготовки к разговору

```text
Смоделируй разговор.
Сначала задай 5 неудобных вопросов, которые может задать собеседник.
Потом помоги подготовить спокойные ответы.
```

### Фишка 28. AI для документов

```text
Проанализируй документ.
Не делай юридический вывод как факт.
Раздели:
1. что прямо написано;
2. что может означать;
3. какие риски;
```

```text
Опиши, что видно на изображении.
```

### Фишка 30. AI для голоса

Голос полезен для:

- языковой практики;

### Фишка 32. AI для файлов

## Антипаттерны

### 1. "Сделай хорошо"

Проблема: нет критерия "хорошо".

Исправление:

```text
Хороший результат должен:
- ...
- ...
- ...
```

### 2. "Расскажи всё"

Проблема: слишком широкая задача.

Исправление:

```text
Мне нужно понять [узкий аспект], чтобы [действие].
```

### 3. "Думай глубже"

Проблема: не задаёт процедуру.

Исправление:

```text
Разбери структуру, допущения, альтернативы, риски и проверку.
```

### 4. Много противоречивых ролей

Плохо:

```text
Будь строгим, мягким, академичным, смешным, кратким, подробным и продающим.
```

Лучше:

```text
Действуй как [одна роль] для [одной задачи].
```

### 5. Огромный контекст без структуры

Проблема: важное теряется.

Исправление: сначала сжать, потом использовать.

### 6. Важные факты без источников

Проблема: модель может галлюцинировать.

Исправление: требовать источники, цитаты, статус вывода и внешнюю проверку.

### 7. Готовые ответы в обучении

Проблема: иллюзия понимания.

Исправление: сначала попытка, потом feedback.

### 8. Слишком длинный чат

Проблема: context drift.

Исправление: summary и новый чат.

### 9. Поиск "лучшей модели" вместо хорошего ТЗ

Проблема: модель меняется, а контекст остаётся слабым.

Исправление: сначала улучшить постановку задачи.

### 10. Автоматизация без копий

Проблема: агент может испортить файлы.

Исправление: копии, Git, план, ограничения, разрешения.

### 11. Доверие к AI в high-stakes задачах

Проблема: риск ошибок в праве, медицине, финансах, безопасности, свежих данных.

Исправление: использовать AI как подготовку, но проверять у источников и специалистов.

### 12. AI-слог без редактуры

Проблема: текст выглядит гладким, но пустым.

Исправление: убрать канцелярит, общие фразы, повторы, искусственную симметрию.

## Проверка фактов и галлюцинаций

AI может:

- придумать источник;
- перепутать дату;
- ошибиться в версии библиотеки;
- неверно интерпретировать документ;
- сгладить неопределённость;
- выдать гипотезу как факт;
- уверенно обобщить частный случай;
- подстроиться под ожидание пользователя.

### Протокол проверки

```text
Проверь ответ.
Раздели утверждения на:
1. подтверждённые факты;
2. выводы из фактов;
3. гипотезы;
4. спорные утверждения;
5. места, где нужен источник;
6. возможные ошибки.

Не добавляй новые факты без пометки.
```

### Когда обязательно проверять актуальность

Проверка нужна для:

- законов;
- судебной практики;
- налогов;
- цен;
- версий библиотек;
- API;
- медицинских рекомендаций;
- финансовых данных;
- новостей;
- правил платформ;
- публичных должностей;
- научных публикаций;
- расписаний;
- рынков труда.

Если у AI нет доступа к свежим источникам, он может дать устаревший ответ.

## Приватность и безопасность

Не отправляйте без необходимости:

- пароли;
- API-ключи;
- приватные документы;
- медицинские данные;
- финансовые данные;
- персональные данные третьих лиц;
- коммерческие секреты;
- исходники закрытых проектов;
- юридически чувствительные материалы.

Если нужно использовать чувствительный материал:

1. Удалить лишние персональные данные.
2. Работать на копиях.
3. Ограничить доступ агента.
4. Проверить настройки хранения данных.
5. Не использовать сторонние скиллы без проверки.
6. Не давать агенту лишние разрешения.

## 7-дневный старт

### День 1. Базовый промпт

Сделать один шаблон:

```text
цель -> контекст -> формат -> критерии -> ограничения -> вопросы
```

Использовать его для 3 задач.

### День 2. Контекст

Для одной сложной задачи сначала наговорить или набросать сырой контекст, попросить AI структурировать его и только потом дать финальный запрос.

### День 3. Критика

Для одного ответа использовать:

```text
структура -> критика -> синтез
```

И отдельно:

```text
убеди меня в обратном, но оцени силу аргументов.
```

### День 4. Источники

Взять один документ и применить режим строгого извлечения:

- только источник;
- пустое лучше ложного;
- основание для каждого вывода;
- статус вывода.

### День 5. Обучение

Использовать AI only after attempt:

1. Своя попытка.
2. AI-feedback.
3. Похожая задача без AI.
4. Короткая запись ошибки.

### День 6. Повторяемый процесс

Найти одну задачу, которую вы повторяете.

Спросить:

```text
Как превратить этот процесс в шаблон или скилл?
```

### День 7. Аудит

Проверить:

- какие промпты реально помогли;
- где AI дал воду;
- где не хватило контекста;
- где нужна проверка фактов;
- что стоит сохранить как шаблон;
- что удалить.

## Минимальная личная система AI

Достаточно 6 элементов:

1. `Паспорт пользователя / проекта`.
2. `Паспорт стиля`.
3. `Универсальный промпт`.
4. `Шаблон строгой проверки фактов`.
5. `Шаблон критики`.
6. `Папка удачных промптов / скиллов`.

Если система становится тяжелее, она может начать мешать. Цель не в коллекции промптов, а в том, чтобы быстрее получать проверяемый результат.

## Готовые шаблоны

### Шаблон 1. Универсальная задача

```text
Задача:

Контекст:

Цель:

Что уже есть:

Ограничения:

Формат:

Критерии качества:

Что нельзя делать:

Если данных недостаточно:
```

### Шаблон 2. Строгая проверка источника

```text
Работай только с предоставленным источником.
Не используй внешние знания.
Если данных недостаточно, пиши "Недостаточно данных".

Для каждого утверждения укажи:
1. утверждение;
2. статус;
3. основание;
4. уровень уверенности;
5. что неизвестно.

Отдельно вынеси противоречия.
```

### Шаблон 3. Критик

```text
Проверь результат как критик.

Найди:
1. слабые места;
2. скрытые допущения;
3. неподтверждённые факты;
4. риски;
5. альтернативы;
6. что нужно проверить.

Оцени силу замечаний.
После критики предложи улучшенную версию.
```

### Шаблон 4. Несколько экспертов

```text
Разбери задачу через 3-5 экспертных ролей.

Для каждой роли:
1. что считает главным;
2. какие риски видит;
3. какие рекомендации даёт;
4. какие ограничения учитывает.

Затем:
1. сравни позиции;
2. покажи противоречия;
3. собери общий план.
```

### Шаблон 5. Редактура текста

```text
Отредактируй текст.

Цель:

Стиль:

Проверь:
1. точность;
2. лишнюю гладкость;
3. канцелярит;
4. общие фразы;
5. неподтверждённые утверждения;
6. тон;
7. структуру.

Не добавляй новых фактов.

Выдай:
1. исправленный текст;
2. список главных правок;
3. что осталось спорным.
```

### Шаблон 6. Обучение с AI

```text
Я изучаю [тема].
Моя цель: [что уметь делать].

Моя попытка:
[текст / код / ответ]

Не решай за меня.
Сделай:
1. найди главную ошибку;
2. задай наводящий вопрос;
3. дай минимальную подсказку;
4. предложи похожую задачу;
5. после моей повторной попытки дай feedback.
```

### Шаблон 7. Code review

```text
Проведи code review.

Контекст:

Код:

Проверь:
1. корректность;
2. edge cases;
3. обработку ошибок;
4. тестируемость;
5. читаемость;
6. архитектурные риски;
7. production-риски.

Формат:
- критичные проблемы;
- важные улучшения;
- мелкие замечания;
- тесты;
- что исправить первым.
```

### Шаблон 8. Агентная задача

```text
Одна задача:

Рабочая папка:

Материалы:

Ожидаемый результат:

Ограничения:
- не менять оригиналы;
- не выходить за папку;
- сначала показать план;
- спрашивать перед опасными действиями.

Критерии успеха:

Сначала проверь понимание и задай вопросы.
```

### Шаблон 9. Создание скилла

```text
Оформи этот рабочий процесс как скилл.

Включи:
1. название;
2. когда использовать;
3. входные данные;
4. пошаговый процесс;
5. критерии качества;
6. типичные ошибки;
7. ограничения безопасности;
8. формат результата;
9. пример хорошего результата.
```

### Шаблон 10. Аудит памяти

```text
Сделай аудит того, что ты знаешь обо мне.

Раздели:
1. подтверждённые факты;
2. предположения;
3. что может быть устаревшим;
4. возможные заблуждения;
5. что стоит уточнить;
6. какие настройки улучшат нашу работу.

Задавай максимум 3 вопроса за раз.
```

## Что особенно важно учесть

1. AI не обязан сам понимать вашу реальную задачу.
2. Первый ответ часто является черновиком, а не итогом.
3. Гладкий текст не равен точности.
4. Для фактов нужны источники.
5. Для навыка нужна самостоятельная попытка.
6. Для агентов нужны ограничения и копии.
7. Для повторяемых задач нужны скиллы.
8. Для длинных чатов нужны summary и новые сессии.
9. Для стиля нужны примеры и редактура.
10. Для безопасности нужны технические барьеры, а не только просьбы.

## Чему не стоит уделять слишком много внимания

Не стоит слишком зацикливаться на:

- поиске "идеального промпта";
- постоянной смене модели;
- длинных библиотеках промптов без применения;
- сложных агентных системах для простых задач;
- deep research там, где нужен короткий ответ;
- автоматизации до понимания процесса;
- скиллах до появления повторяемой задачи;
- красивых AI-ответах без проверки;
- впечатляющих демо без переноса в вашу работу.

## Уровень надёжности практик

| Практика | Оценка |
|---|---|
| Контекст + цель + формат + критерии | сильная практическая основа |
| Отделение фактов от гипотез | сильная основа для критического мышления |
| Заземление в источниках | сильная практическая защита от выдумок, но не гарантия |
| Пустое поле лучше ложного | особенно полезно для извлечения данных |
| AI-feedback после попытки | умеренно сильная учебная логика |
| AI как готовый исполнитель в обучении | рискованно |
| Метод нескольких экспертов | полезная аналитическая эвристика, не замена экспертизы |
| "Убеди меня в обратном" | полезно для смены перспективы, не фактчек |
| Память ChatGPT | полезно для персонализации, требует аудита |
| Скиллы | полезны для повторяемых процессов |
| Агенты | сильны для многошаговых задач, требуют безопасности |
| Хуки и технические ограничения | важны для критичных правил |

## Итоговый протокол

```text
1. Назвать результат.
2. Дать контекст.
3. Задать формат.
4. Задать критерии качества.
5. Попросить вопросы, если данных мало.
6. Получить черновик.
7. Запустить критику.
8. Проверить факты.
9. Исправить.
10. Сохранить удачный шаблон.
11. Если задача повторяется - сделать скилл.
12. Если задача опасная - добавить технические ограничения.
```
