# Промты, сценарии использования - инструкция пользователя

## Статус и источники

Инструкция собрана по локальному Obsidian-хранилищу. Было просмотрено 4687 Markdown-файлов без `AGENTS.md`; затем отобраны файлы, где явно встречаются промпты, сценарии использования AI, ChatGPT, Claude, NotebookLM, Codex, ИИ-тьюторы, AI-feedback, работа с источниками, обучение, код, документы, тексты и личная организация.

Основной корпус источников: `AI`, `Learning`, `Memory`, `4sort`. Дополнительно использованы релевантные материалы из `IT`, `Job`, `Business`, `Voice`, `Scribbles`.

Цель этой инструкции - не собрать все фразы подряд, а дать рабочую систему: какой промпт использовать в какой ситуации, как его адаптировать, где он может сломаться и как проверять результат.

## Главная мысль

Хороший промпт - это не магическая фраза. Это короткая постановка задачи с контекстом, входными данными, форматом результата, ограничениями и критерием качества.

Слабый запрос выглядит так:

```text
Объясни мне эту тему.
```

Сильный запрос выглядит так:

```text
Я изучаю тему: [тема].
Мой уровень: [что уже знаю].
Моя цель: [что хочу уметь делать].

Не давай длинную лекцию.
Сначала проверь моё понимание: задай 3 вопроса по одному.
После моих ответов укажи:
1. что верно;
2. где пробел;
3. какую одну практическую задачу мне сделать дальше.

Если данных недостаточно, задай один уточняющий вопрос.
```

Разница не в длине. Разница в управлении работой модели.

---

## Базовая модель промпта

Используй эту формулу как основу:

```text
Цель:
[какой результат нужен]

Контекст:
[ситуация, исходные данные, уровень, ограничения, аудитория]

Задача:
[одно конкретное действие]

Формат ответа:
[таблица, список, чек-лист, план, JSON, конспект, письмо, код, вопросы]

Критерии качества:
[как понять, что ответ хороший]

Ограничения:
[что не делать, какие источники использовать, что не выдумывать]

Если данных недостаточно:
[задать вопрос / отметить неизвестное / не делать вывод]
```

Мини-версия:

```text
Нужен результат: [результат].
Контекст: [контекст].
Сделай: [действие].
Формат: [формат].
Критерии: [критерии качества].
Если данных недостаточно, не выдумывай - укажи, чего не хватает.
```

## Как выбирать промпт под задачу

| Если нужно | Используй сценарий |
|---|---|
| Быстро получить точный ответ | Жёсткий формат ответа |
| Не дать модели уйти в воду | Таблица, лимит пунктов, "без текста вне формата" |
| Проверить источники | Аудит источников |
| Извлечь факты из документа | Режим строгого извлечения |
| Проверить своё понимание | AI-feedback после собственной попытки |
| Учиться без готовых ответов | Сократический тьютор |
| Подготовиться к экзамену | Вопросы, блиц-опрос, задачи, карточки |
| Принять решение | Матрица выбора + риски + условия |
| Найти слабые места | Критик / "убеди меня в обратном" |
| Получить объёмную оценку | Несколько экспертных ролей |
| Улучшить плохой ответ | Заточка или фильтр-проход |
| Написать код с AI | Сначала план, потом код, потом тесты |
| Работать с агентом | To-do, границы, критерий готовности, проверка diff |
| Сохранить контекст | Контекстный конспект для нового чата |
| Сделать текст менее AI-подобным | Редактура против AI-слога |
| Создать изображение | Subject, action, environment, style, composition, lighting, negative |
| Создать голос/озвучку | Тембр, возраст, темп, громкость, эмоциональность, negative prompt |

# 1. Универсальные промпты

## 1.1. Универсальный рабочий промпт

Когда использовать: почти для любой задачи, где нужен качественный результат, а не быстрый черновик.

```text
Действуй как [роль].

Контекст:
[опиши ситуацию, исходные данные, что уже известно, что пробовали]

Моя цель:
[какой практический результат нужен]

Аудитория:
[для кого нужен результат и какой у аудитории уровень]

Задача:
[что конкретно нужно сделать]

Формат ответа:
[таблица / список / чек-лист / инструкция / план / код / письмо / конспект]

Ограничения:
[чего не делать, какие рамки соблюдать]

Критерии качества:
[как понять, что результат хороший]

Если данных недостаточно:
сначала перечисли, чего не хватает;
затем дай лучший возможный ответ на основе имеющегося.
```

Вариации:

- Для короткого ответа: "Ответь максимум в 7 пунктах, каждый пункт - одно предложение".
- Для строгой работы: "Отделяй факты, предположения и рекомендации".
- Для решения: "Начни с вывода, затем дай причины, риски и следующие шаги".
- Для обучения: "Не давай готовое решение, пока я не сделаю попытку".

Критерий качества: модель не угадывает задачу, а работает в заданных границах.

---

## 1.2. Промпт с уточняющими вопросами

Когда использовать: когда задача неясная, цена ошибки выше обычной или входных данных мало.

```text
Перед тем как отвечать, оцени, достаточно ли данных для качественного ответа.

Если данных недостаточно:
1. задай до 3 уточняющих вопросов;
2. объясни, зачем нужен каждый вопрос;
3. не давай основной ответ до моих уточнений.

Если данных достаточно:
ответь в формате:
- краткий вывод;
- аргументы;
- риски;
- что делать дальше.
```

Более жёсткая версия:

```text
Прежде чем отвечать, оцени неопределённость.
Если неопределённость высокая, задай уточняющие вопросы.
Не заполняй пробелы догадками.
```

Фишка: не требуй у модели точный числовой "процент неопределённости" как научный факт. Лучше использовать практический критерий: "данных достаточно / недостаточно / есть спорные места".

---

## 1.3. Outcome-first prompt

Когда использовать: для сильных моделей, агентов, сложных задач, где важен результат, а не буквальное следование микрошагам.

```text
Нужен результат:
[описание результата]

Хороший результат должен:
- [критерий 1];
- [критерий 2];
- [критерий 3].

Границы задачи:
- используй только [источники / файлы / данные];
- не расширяй задачу без просьбы;
- не делай [запрет].

Условие остановки:
остановись, когда [конкретный критерий завершения].

Если есть несколько путей решения, выбери самый надёжный и кратко объясни выбор.
```

Зачем: длинные процессные инструкции иногда мешают. Лучше описать результат, критерии и границы.

Где особенно полезно:

- анализ документов;
- кодовые задачи;
- подготовка инструкций;
- работа агента;
- структурирование большого материала.

---

## 1.4. Диагностический промпт перед основной задачей

Когда использовать: до переписывания текста, кода, документа, учебного плана.

```text
Пока не исправляй и не переписывай.

Сначала проанализируй входные данные и выдели:
1. что это за тип материала;
2. главную цель;
3. скрытые ограничения;
4. риски ошибки;
5. чего не хватает;
6. какой формат результата лучше выбрать.

После диагностики предложи план работы.
```

Для кода:

```text
Проанализируй код.
Пока не переписывай.

Определи:
1. язык и стек;
2. стиль проекта;
3. что код пытается сделать;
4. возможные точки поломки;
5. ограничения, которые нужно соблюдать;
6. какие тесты нужны перед изменением.
```

Фишка: "пока не переписывай" сохраняет ошибки как диагностическую информацию.

---

## 1.5. Промпт для улучшения самого промпта

Когда использовать: если запрос кажется мутным, модель отвечает не туда или задача повторяется часто.

```text
Улучши мой промпт.

Исходный промпт:
"""
[вставить промпт]
"""

Моя цель:
[какой результат я хочу получать]

Сделай:
1. найди слабые места промпта;
2. укажи, где не хватает контекста;
3. убери противоречия и лишние инструкции;
4. добавь формат ответа и критерии качества;
5. дай короткую и полную версию промпта;
6. объясни, в каких задачах каждая версия лучше.
```

Проверочная версия:

```text
Проверь этот промпт на риск плохого результата.

Найди:
- двусмысленности;
- скрытые требования;
- конфликтующие инструкции;
- слишком широкую задачу;
- отсутствие формата;
- отсутствие критериев качества;
- риск галлюцинаций.

Затем дай исправленную версию.
```

# 2. Управление форматом и краткостью

## 2.1. Жёсткий формат ответа

Когда использовать: если модель пишет слишком много.

```text
Дай ответ строго в формате:

1. Краткий вывод - 1 предложение.
2. Почему - 3 пункта.
3. Что делать дальше - 3 шага.
4. Главный риск - 1 пункт.

Без вступления, без общих рассуждений, без текста после списка.
```

Для сравнения:

```text
Сравни вариант A и вариант B в таблице.

Столбцы:
- критерий;
- вариант A;
- вариант B;
- какой вариант лучше;
- почему.

Критерии:
- стоимость;
- скорость;
- риски;
- сложность внедрения;
- долгосрочная польза.

Без текста вне таблицы.
```

Фишка: "будь кратким" работает хуже, чем измеримый формат.

---

## 2.2. Заточка длинного ответа

Когда использовать: ответ полезный, но слишком расплывчатый.

```text
Сократи предыдущий ответ на 50%.

Оставь только:
- факты;
- выводы;
- конкретные действия;
- риски.

Убери:
- вступления;
- повторы;
- общие рассуждения;
- очевидные советы;
- осторожные формулировки без практической пользы.

Не добавляй новые идеи.
```

Вариации:

- "Сократи на 30%, сохрани структуру".
- "Сократи до 10 пунктов".
- "Оставь только то, что влияет на решение".
- "Убери всё, что нельзя применить сегодня".

Важно: фраза "не добавляй новые идеи" снижает риск, что модель начнёт заново сочинять текст.

---

## 2.3. Фильтр-проход в новом чате

Когда использовать: длинный чат загрязнился, модель повторяется или смешивает старые версии.

```text
Ты - фильтр полезности.

Я вставлю длинный материал.
Не пересказывай его полностью.

Извлеки только:
1. главные выводы;
2. факты;
3. решения;
4. риски;
5. открытые вопросы;
6. следующие действия.

Убери повторы, воду, эмоциональные формулировки и несущественные детали.
Не добавляй новые идеи.

Материал:
"""
[вставить текст]
"""
```

Фишка: новый чат помогает не тащить за собой старые ошибки контекста.

---

## 2.4. Контекстный конспект для переноса в новый чат

Когда использовать: длинная задача продолжается, контекст переполнен, но нужно сохранить смысл.

```text
Сделай компактный контекстный конспект этого чата для переноса в новый чат.

Включи:
1. цель работы;
2. ключевые факты;
3. принятые решения;
4. мои предпочтения по стилю и формату;
5. важные ограничения;
6. незавершённые задачи;
7. формулировки, которые нужно сохранить;
8. что не нужно повторять.

Не добавляй новых выводов.
Пиши так, чтобы другой чат мог продолжить работу без потери контекста.
```

# 3. Источники, факты и антигаллюцинации

## 3.1. Режим строгого извлечения фактов

Когда использовать: договоры, исследования, медицинские выписки, юридические тексты, таблицы, расшифровки, документы.

```text
Работай только с предоставленным источником.

Запрещено:
- добавлять внешние сведения;
- достраивать причинно-следственные связи;
- предполагать мотивы;
- делать выводы, если они прямо не следуют из текста;
- превращать вероятные интерпретации в установленные факты.

Для каждого утверждения укажи:
1. Само утверждение.
2. Статус:
   - прямо указано;
   - следует из явно указанных фактов;
   - вероятная интерпретация;
   - недостаточно данных.
3. Основание: короткий пересказ конкретного фрагмента.
4. Уровень уверенности: высокий / средний / низкий.
5. Что осталось неизвестным.

Если данных недостаточно, прямо напиши:
"На основании предоставленного материала это установить нельзя".
```

Фишка: неверный ответ хуже пустого. Разрешай модели писать "недостаточно данных".

---

## 3.2. Аудит источников

Когда использовать: перед глубоким разбором темы, особенно если источников много.

Промпт 1 - достоверность:

```text
Проанализируй все источники и составь таблицу аудита:
- источник;
- краткое описание;
- год / дата;
- тип источника;
- насколько можно доверять;
- ограничения;
- стоит ли использовать дальше.
```

Промпт 2 - противоречия:

```text
Посмотри все источники.
Найди места, где они противоречат друг другу или расходятся в оценках.

Указывай только реальные расхождения.
Для каждого расхождения дай:
- тема расхождения;
- источник A;
- источник B;
- в чём отличие;
- что нужно проверить вручную.
```

Промпт 3 - слепые зоны:

```text
Какие важные взгляды, школы, источники или менее распространённые позиции отсутствуют в моей подборке?

Для каждой слепой зоны укажи:
- что отсутствует;
- почему это может быть важно;
- где стоит поискать;
- как это может изменить выводы.
```

---

## 3.3. Разделение фактов, гипотез и преувеличений

Когда использовать: научпоп, обучение, здоровье, психология, продуктивность, AI-обещания.

```text
Проверь материал с научной осторожностью.

Раздели:
1. Что хорошо подтверждено.
2. Что правдоподобно, но зависит от контекста.
3. Что является гипотезой.
4. Что похоже на популярное упрощение.
5. Какие выводы были бы чрезмерными.

Отдельно укажи:
- какие термины нужно определить точнее;
- какие источники стоит проверить;
- где автор делает слишком сильный вывод;
- как применить идею без переобобщения.
```

---

## 3.4. Проверка ответа самой моделью

Когда использовать: после важного ответа.

```text
Проверь свой ответ по критериям:

1. Соответствие задаче.
2. Соответствие формату.
3. Конкретность.
4. Проверяемость фактов.
5. Отделение фактов от предположений.
6. Практическая применимость.
7. Отсутствие противоречий.
8. Что можно улучшить.

После проверки:
- перечисли найденные проблемы;
- дай исправленную версию ответа.
```

Ограничение: самопроверка модели не заменяет фактчек. Она помогает найти явные дефекты, но не гарантирует истинность.

# 4. Обучение с AI

## 4.1. Главный учебный режим: сначала попытка, потом AI-feedback

Когда использовать: программирование, языки, экзамены, чтение сложных текстов, аналитическое мышление.

```text
Я изучаю [тема].
Не пиши готовое решение сразу.

Моя попытка:
[объяснение / код / решение / план]

Мой прогноз:
[что, по-моему, должно получиться]

Факт / ошибка:
[что произошло]

Сделай:
1. Найди одну главную ошибку в рассуждении.
2. Дай подсказку, но не полный ответ.
3. Сформулируй один вопрос для active recall.
4. Предложи одну похожую задачу для повторной попытки.
```

Зачем: AI должен усиливать цикл "прогноз -> попытка -> ошибка -> feedback -> повтор", а не заменять попытку готовым ответом.

---

## 4.2. Сократический тьютор

Когда использовать: если нужно понять тему, а не получить пересказ.

```text
Я изучаю [тема].
Работай как сократический тьютор.

Правила:
1. Не давай готовый ответ сразу.
2. Сначала выясни, что я уже понимаю.
3. Задавай по одному вопросу за раз.
4. Если я ошибаюсь, не исправляй сразу полностью - покажи, где противоречие.
5. Проси меня предсказать результат или объяснить механизм.
6. После моей попытки дай короткую обратную связь:
   - что верно;
   - где ошибка;
   - какая ментальная модель нужна.
7. В конце дай маленькое упражнение без подсказок.
8. Только после моей попытки покажи эталонный вариант.

Цель: сформировать понимание, которое я смогу применить без ИИ.
```

---

## 4.3. Проверка понимания

Когда использовать: после чтения, лекции, конспекта, видео, главы учебника.

```text
Проверь моё понимание темы [тема].

Вот как я это понимаю:
"""
[моё объяснение]
"""

Сделай:
1. Что в объяснении верно.
2. Что неточно, неполно или спорно.
3. Какие термины я использую слишком туманно.
4. Где может быть иллюзия понимания.
5. Какие важные элементы темы я пропустил.
6. Какие 5 вопросов проверят моё понимание.
7. Какая маленькая практическая задача покажет, могу ли я применить тему.

Не переписывай мой текст ради красоты.
Главная цель - проверить модель понимания.
```

## 4.4. Промпт против самообмана

Когда использовать: когда кажется, что тема понятна, но есть риск "узнавания" вместо знания.

```text
Проверь моё объяснение на самообман.

Найди места, где:
- формулировка звучит убедительно, но не доказывает понимание;
- я использую общие слова вместо механизма;
- я пропускаю важный шаг рассуждения;
- я путаю похожие понятия;
- я не могу привести пример;
- я делаю вывод шире, чем позволяют данные.

После этого дай:
1. главный пробел;
2. один вопрос для проверки;
3. одну маленькую задачу на 15-30 минут.
```

## 4.5. Генерация вопросов для active recall

Когда использовать: вместо перечитывания конспекта.

```text
Составь вопросы для проверки понимания темы: [тема].

Нужно 12 вопросов:
- 2 на базовое определение;
- 2 на механизм;
- 2 на отличие от похожих понятий;
- 2 на практическое применение;
- 2 на типичные ошибки;
- 1 на ограничения;
- 1 на перенос в новый контекст.

Не показывай ответы сразу.
Задавай вопросы по одному.
После моего ответа оцени по шкале:
0 - не понял;
1 - частично;
2 - в целом верно, но есть пробелы;
3 - хорошо;
4 - могу применять.
```

## 4.6. Практическая задача вместо пассивного объяснения

Когда использовать: если тема должна перейти в навык.

```text
Дай мне одну практическую задачу по теме: [тема].

Требования:
- задача должна быть небольшой;
- выполнение - 15-30 минут;
- она должна проверять понимание, а не механическое копирование;
- не показывай решение сразу;
- после моей попытки дай обратную связь и одну вариацию задачи.
```

Лестница практики:

```text
Сделай лестницу практики по теме [тема].

Нужно 5 задач:
1. совсем простая;
2. базовая;
3. с типичной ошибкой;
4. с переносом в новый контекст;
5. мини-проект или реальный кейс.

Для каждой задачи укажи:
- что она проверяет;
- какой тип ошибки вероятен;
- критерий успешного выполнения.

Решения не показывай, пока я не попрошу.
```

# 5. Мышление, решения и критика

## 5.1. Решение через матрицу

Когда использовать: выбор технологии, работы, курса, покупки, стратегии.

```text
Помоги принять решение между вариантами:
- A: [описание]
- B: [описание]
- C: [описание]

Контекст:
[цель, ограничения, сроки, ресурсы]

Сравни в таблице по критериям:
- польза;
- стоимость;
- риск;
- сложность внедрения;
- обратимость решения;
- долгосрочные последствия;
- что может пойти не так.

После таблицы дай:
1. лучший вариант при текущих вводных;
2. при каких условиях выбор изменится;
3. какой минимальный эксперимент провести за 7 дней.
```

---

## 5.2. Критик / "убеди меня в обратном"

Когда использовать: когда есть позиция, план, решение, текст, гипотеза.

Простая версия:

```text
Убеди меня в обратном.
```

Лучше:

```text
Проверь свой предыдущий ответ критически.

Найди:
1. слабые места;
2. спорные допущения;
3. ситуации, где вывод не работает;
4. аргументы против;
5. что нужно проверить фактами.

Не спорь ради спора.
Оцени силу каждого контраргумента.
```

Трёхшаговая схема:

```text
Шаг 1. Дай свою основную позицию по этому вопросу.
Кратко объясни логику.
```

```text
Шаг 2. Теперь разбери позицию как критик.
Найди слабые места и аргументы против.
```

```text
Шаг 3. Сравни обе позиции.
Какая сильнее и при каких условиях?
Отдели факты от предположений.
Дай итоговый вывод.
```

Ограничение: это не фактчек. Модель может придумать убедительные, но слабые контраргументы.

---

## 5.3. Метод нескольких экспертов

Когда использовать: сложная задача с несколькими перспективами: код, обучение, бизнес, юридическая позиция, карьера, продукт.

```text
Проанализируй мою задачу методом нескольких экспертных перспектив.

Задача:
[вставить задачу]

Сделай разбор в 4 этапа:

1. Определи 3-5 экспертных ролей, которые лучше всего подходят для этой задачи.
   Для каждой роли кратко объясни, почему она здесь нужна.

2. Дай анализ с позиции каждой роли:
   - что она считает главным;
   - какие проблемы или риски видит;
   - какие решения предлагает;
   - какие ограничения учитывает.

3. Сравни позиции экспертов:
   - где они согласны;
   - где расходятся;
   - какие противоречия важно учесть.

4. Собери итоговый вывод:
   - что действительно важно;
   - какие действия выполнить в первую очередь;
   - какие ошибки не допустить;
   - какой практический план использовать.

Отделяй факты от предположений.
Не выдумывай недостающие данные.
Если информации недостаточно, сначала явно укажи, чего не хватает.
```

Короткая версия:

```text
Разбери эту задачу через 4 экспертные роли.

1. Сам выбери роли, которые лучше всего подходят.
2. Покажи взгляд каждой роли.
3. Укажи, где их позиции совпадают и где отличаются.
4. Собери общий практический вывод и план действий.

Задача:
[вставить задачу]
```

## 5.4. Pre-mortem: что сломает план

Когда использовать: перед запуском проекта, обучающего плана, продукта, важного письма, юридического действия.

```text
4. какой риск самый вероятный;
5. какой риск самый опасный;
6. что проверить в первую очередь.

Не драматизируй. Оцени реалистично.
```

# 6. Тексты, документы и контент

## 6.1. Редактура против AI-слога

Когда использовать: если текст гладкий, общий, канцелярский или похож на универсальную справку.

```text
Отредактируй текст так, чтобы убрать типичные признаки AI-слога.

Текст:
"""
[вставить текст]
"""

Что исправить:
1. Убери пустые вводные фразы: "важно отметить", "следует подчеркнуть", "играет ключевую роль", "является важным аспектом".
2. Уменьши канцелярит и чрезмерно гладкий стиль.
3. Убери синонимическую карусель и ненужные повторы.
4. Замени общие фразы на конкретные формулировки.
5. Сделай предложения разной длины, но без искусственной "живости".
6. Убери чрезмерную дидактичность и тон универсальной справки.
7. Не добавляй фактов, которых нет в исходном тексте.
8. Сохрани смысл, структуру и спокойный профессиональный тон.

Формат ответа:
1. Сначала дай отредактированную версию.
2. Затем кратко перечисли, что именно было исправлено.
```

Мини-версия:

```text
Отредактируй текст: убери AI-слог, канцелярит, пустые вводные, чрезмерную гладкость и общие фразы.
Сохрани смысл, не добавляй новых фактов, сделай текст спокойным, точным и человеческим.

Текст:
"""
[вставить текст]
"""
```

## 6.2. Анализ текста перед переписыванием

```text
- где формулировка слишком сильная;
- итоговая рекомендация.

Каждый пункт - коротко. Без вступления.

Текст:
"""
[вставить текст]
"""
```


## 6.4. Сценарий видео / ytb-формат

# 7. Код, разработка и AI-агенты

## 7.1. Разбор ошибки в коде

````text

Контекст:
- язык: [язык];
- стек: [стек];
- что я пытаюсь сделать: [цель];

````

```text
Сначала дай подсказку без готового кода.
Полное решение показывай только после моей попытки.
```

````text

4. архитектурные риски;
````

## 7.3. Сначала архитектура, потом код

Когда использовать: новая функция, сервис, модуль, интеграция.

```text

[описание]

Пока не пиши код.

Сначала предложи:

2. какие файлы или модули затронуть;
3. какие типы / структуры нужны;
4. какие edge cases учесть;
5. какие тесты написать;
6. риски безопасности;
7. критерий готовности.

После этого остановись и жди моего подтверждения.
```

## 7.4. Три варианта решения

Когда использовать: архитектура, рефакторинг, выбор библиотеки, проектирование API.

```text
Предложи 3 варианта решения задачи.

[описание]

- риски;
```

## 7.6. Go-промпты

```text
Проанализируй этот Go-код на:
- data races;
- deadlocks;
- утечки goroutines;
- блокировки по каналам;
- неправильную работу context.

Для каждой проблемы дай:
1. где она возникает;
2. как проявится в production;
3. как исправить;
4. какой тест это поймает.
```

Anti-cheat для обучения:

```text
Объясни решение, не предоставляя готового кода.
Только концепции, шаги и ключевые идеи.
После этого дай маленькое упражнение.
```

## 7.7. Работа с AI-агентом

Когда использовать: Codex, Claude Code, Copilot Agent и другие инструменты, которые читают и меняют файлы.

```text
Работай как AI-агент в репозитории.

Задача:
[задача]

Границы:
- работай только в [папка / файлы];
- не меняй [запрещённые файлы];
- не делай рефакторинг вне задачи;
- не добавляй зависимости без объяснения.

Перед изменениями:
1. прочитай релевантные файлы;
2. составь краткий план;
3. укажи риски;
4. дождись подтверждения, если изменение затрагивает больше [N] файлов.

Критерий готовности:
- [что должно работать];
- [какие тесты запустить];
- [какой результат ожидается].

После работы:
1. перечисли изменённые файлы;
2. объясни, что изменено;
3. укажи, какие проверки прошли;
4. что осталось рискованным.
```

Фишки для агентов:

- Одна задача - одна сессия.
- Перед началом проси to-do.
- Ограничивай папки.
- Проверяй diff.
- Не доверяй авторизацию и безопасность по умолчанию.
- Если агент пошёл не туда, останавливай и уточняй.
- После промежуточного результата сохраняй состояние и начинай новую сессию.

# 14. Фишки, которые чаще всего улучшают результат

## Фишка 1. "Сначала спроси"

```text
Если для ответа не хватает данных, задай один уточняющий вопрос и остановись.
```

## Фишка 2. "Без текста вне формата"

```text
Ответь только таблицей. Без вступления и текста после таблицы.
```

## Фишка 3. "Неправильный хуже пустого"

```text
Если данных недостаточно, лучше напиши "недостаточно данных", чем предполагай.
```

## Фишка 4. "Тип извлечения"

```text
Для каждого факта укажи тип: дословно / переформулировано / выведено из явно указанных фактов / предположение.
```

## Фишка 5. "Пока не переписывай"

```text
Пока не исправляй. Сначала покажи, где моя модель сломалась.
```

## Фишка 6. "Три варианта"

```text
Дай 3 варианта решения с плюсами, минусами, рисками и условиями применения.
```

## Фишка 7. "Условие остановки"

```text
Остановись, когда дашь 5 применимых шагов. Не расширяй задачу.
```

## Фишка 8. "Фильтр полезности"

```text
Извлеки только факты, выводы, риски и действия. Убери повторы и воду.
```

## Фишка 9. "Собственная позиция до AI"

```text
Сначала я дам свою позицию. Потом ты проверишь её на слабые места.
```

## Фишка 10. "Не искусственный баланс"

```text
Не создавай искусственный баланс, если одна позиция явно сильнее.
```

## Фишка 11. "Примеры вместо описаний"

```text
Вот 2 примера результата, который мне нравится:
[пример 1]
[пример 2]

Сделай новый результат в той же логике, но не копируй дословно.
```

## Фишка 12. "Роль плюс критерии, а не только роль"

Плохо:

```text
Ты эксперт. Объясни тему.
```

Лучше:

```text
Действуй как преподаватель-наставник.
Цель - проверить моё понимание и дать следующую задачу.
Критерий качества - я смогу воспроизвести тему без подсказки.
```

# 15. Антипаттерны

| Антипаттерн | Почему плохо | Как исправить |
|---|---|---|
| "Расскажи всё про X" | Слишком широко | Указать цель, уровень и формат |
| "Сделай красиво" | Нет критерия красоты | Дать аудиторию, стиль, референсы |
| "Напиши лучше" | Непонятно, что улучшать | Указать: точность, краткость, структура, тон |
| Огромный контекст без структуры | Главное теряется | Сжать контекст в блоки |
| Противоречивые роли | Модель получает конфликтующие сигналы | Оставить 1-2 совместимые роли |
| "Думай глубже" | Не задаёт процедуры | Дать критерии анализа |
| Просить готовый ответ до попытки | Создаёт иллюзию понимания | Сначала попытка, потом feedback |
| Просить факты без источников | Риск галлюцинаций | Требовать основание и статус утверждений |
| Слепо принимать код | Потеря контроля | Читать diff, запускать тесты, ревьюить |
| Бесконечно уточнять | Можно застрять | Ввести критерий остановки |

Каждый удачный промпт сохраняй как сценарий с примерами и ограничениями.

# 19. Мини-вывод

Рабочая библиотека промптов строится не вокруг "самых сильных фраз", а вокруг повторяемых сценариев.

Лучший промпт отвечает на шесть вопросов:

1. Какой результат нужен?
2. В каком контексте?
3. Что именно сделать?
4. В каком формате?
5. Как проверить качество?
6. Что нельзя выдумывать или расширять?

Если задача важная, добавляй ещё три блока:

1. что делать при нехватке данных;
2. как отделять факты от предположений;
3. когда остановиться.

---
