# Самообучение и учебная архитектура - инструкция пользователя

Режим: `ytb`

Тема: как построить личную систему самообучения: учебная программа, расписание, компетенции, циклы практики, обратная связь, недельная калибровка, публичное обучение, lifelong learning и смена учебной траектории.

Основные источники: `Learning`, `Memory`, `4sort`.

---

## 0. Главная идея

Самообучение ломается не только из-за лени.

Чаще оно ломается из-за плохой архитектуры:

- нет ясной цели;
- слишком много источников;
- темы сформулированы расплывчато;
- нет проверяемых компетенций;
- нет расписания;
- нет самостоятельных попыток;
- нет обратной связи;
- нет недельной калибровки;
- прогресс измеряется часами, а не доказательствами навыка;
- план разрастается быстрее, чем человек успевает учиться.

Рабочая учебная архитектура выглядит так:

```text
цель -> компетенции -> источники -> задания -> учебные циклы -> feedback -> доказательства -> недельная калибровка -> следующий цикл
```

Короткая версия:

```text
Не "что я изучил?", а "что я теперь могу сделать и чем это доказано?"
```

## 1. Карта темы

| Технология | Краткое описание | Главная польза |
| --- | --- | --- |
| Personal Curriculum / личная учебная программа | Личный учебный сезон на 2-12 недель: тема, источники, задания, контрольные точки, итоговый артефакт | Убирает хаос и превращает интерес в проверяемый цикл |
| Личный учебный план | Конкретный план обучения: что, когда, по каким материалам, с какими проверками | Снимает ежедневную нагрузку выбора |
| Самообучение / самообразование | Самостоятельное освоение навыка или области без внешней программы | Даёт автономию, но требует системы контроля |
| Self-regulated learning / SRL | Саморегулируемое обучение: цель, план, мониторинг, коррекция | Помогает управлять обучением как процессом |
| Недельная калибровка | Раз в неделю смотреть, что работает, что убрать, что тренировать дальше | Защищает систему от разрастания |
| Расписание самообучения | Реалистичное распределение учебных блоков с буферами и отдыхом | Делает обучение регулярным |
| Учебные циклы | Короткие циклы: прогноз, попытка, feedback, повтор | Переводят знания в действие |
| Компетенция вместо темы | Учиться не "теме", а проверяемому умению | Делает прогресс наблюдаемым |
| Competency-Based Education / CBE | Обучение через доказанное владение компетенциями | Сдвигает фокус с часов на результат |
| Mastery learning | Двигаться дальше после достижения критерия владения, а не после фиксированного времени | Уменьшает пробелы в базе |
| Learning in Public | Учиться публично: делиться процессом, ошибками, проектами, выводами | Даёт структуру, обратную связь и портфолио |
| Lifelong learning | Непрерывное обучение в течение жизни | Поддерживает профессиональную устойчивость |
| Mindshift | Смена учебной или профессиональной траектории во взрослом возрасте | Помогает менять направление реалистично |

## 2. Самая важная рамка

Учебная архитектура должна отвечать на 7 вопросов.

```text
1. Зачем я это изучаю?
2. Какую компетенцию хочу получить?
3. Какой минимальный источник нужен?
4. Какое задание покажет понимание?
5. Как получу feedback?
6. Какой артефакт останется?
7. Когда пересмотрю план?
```

Если хотя бы три ответа отсутствуют, обучение легко превращается в поток материалов.

---

## 3. Уровни учебной архитектуры

| Уровень | Горизонт | Главный вопрос | Пример |
| --- | --- | --- | --- |
| Траектория | 6-24 месяца | Куда я двигаюсь? | backend, язык, математика, новая профессия |
| Учебный сезон | 4-12 недель | Какой крупный блок закрываю? | основы Go, SQL, system design |
| Недельный фокус | 7 дней | Какая компетенция недели? | писать handler tests |
| Учебный блок | 25-90 минут | Какой цикл закрываю сейчас? | прогноз -> попытка -> feedback |
| Exit ticket | 2-3 минуты | Что понял, что неясно, следующий шаг? | "ошибка в receiver, повторить завтра" |

Хорошая система связывает эти уровни. Плохая система живет только на уровне "сегодня посмотрю что-нибудь полезное".

---

## 4. Что считать прогрессом

Плохие метрики:

- посмотрел видео;
- прочитал главу;
- сохранил ссылку;
- сделал красивый план;
- провел 3 часа за учебой;
- накопил 100 заметок;
- прошел курс до конца.

Эти действия могут быть полезны, но сами по себе не доказывают обучение.

Хорошие метрики:

- решил задачу без подсказки;
- объяснил тему своими словами;
- нашел и исправил ошибку;
- сделал повторную похожую задачу;
- получил обратную связь;
- создал артефакт;
- применил идею в новом контексте;
- понял, что нужно убрать из плана;
- показал доказательство компетенции.

Формула:

```text
активность - это что я делал;
прогресс - это что теперь могу делать лучше.
```

# Часть I. Personal Curriculum / личная учебная программа

## 5. Что такое Personal Curriculum

Personal Curriculum / личная учебная программа - это ограниченный учебный цикл, который взрослый человек проектирует сам.

В нем есть:

- тема;
- цель;
- источники;
- задания;
- контрольные точки;
- обратная связь;
- итоговый артефакт;
- критерий завершения.

Это не отдельная научно доказанная "нейрометодика". Корректнее: это организационная оболочка, которая хорошо согласуется с известными принципами эффективного обучения.

Сила не в названии, а в структуре:

```text
тема -> цель -> материалы -> задания -> проверка -> feedback -> артефакт
```

## 6. Когда личная учебная программа особенно полезна

1. Когда тема большая и легко утонуть.
2. Когда нет внешнего курса.
3. Когда есть много источников, но мало практики.
4. Когда нужно сменить профессию или траекторию.
5. Когда хочется учиться регулярно, но не хватает структуры.
6. Когда есть риск бесконечно смотреть видео.
7. Когда нужно собрать портфолио.
8. Когда нужно превратить интерес в результат.

## 7. Минимальная структура Personal Curriculum

```text
Тема:
Зачем:
Срок:
Ограничение:
Источники:
Компетенции:
Задания:
Feedback:
Итоговый артефакт:
Критерий завершения:
Недельная калибровка:
```

Главное - не список источников, а задания и доказательства.

Если программа состоит только из книг, видео и курсов, это пока не учебная программа. Это список материалов.

## 8. Личная программа на 4 недели

Пример универсальной структуры:

| Неделя | Фокус | Задание | Доказательство |
| --- | --- | --- | --- |
| 1 | Карта темы | Сделать обзор и список компетенций | Mind map, список пробелов |
| 2 | База | Разобрать 2-3 ключевых понятия | Объяснение, мини-тест, схема |
| 3 | Практика | Сделать задачу или мини-проект | Рабочий пример, ошибка, исправление |
| 4 | Сборка | Создать итоговый артефакт | README, конспект, доклад, проект |

## 9. Вариации Personal Curriculum

### 2-недельный спринт

Подходит, если тема узкая.

```text
Неделя 1: карта + базовая практика.
Неделя 2: проверка + артефакт.
```

Пример:

```text
Тема: Docker для Go-сервиса.
Артефакт: Dockerfile, docker-compose, README с типичными ошибками.
```

### 4-недельный учебный сезон

Подходит для средней темы.

```text
1 неделя - обзор.
2 неделя - базовые задания.
3 неделя - перенос и ошибки.
4 неделя - итоговый проект.
```

### 12-недельный курс для себя

Подходит для большой области.

Важно: 12 недель должны быть разделены на модули. Иначе план станет слишком тяжелым.

```text
3 блока по 4 недели:
1. База.
2. Практика.
3. Проект и портфолио.
```

### Deep reading curriculum

Фокус на глубоком чтении.

Структура:

- 1-2 основные книги;
- вопросы к каждой главе;
- конспекты;
- пересказ;
- эссе или обзор;
- обсуждение или публичная заметка.

Риск: читать без практики.

### Project-based curriculum

Фокус на проекте.

Структура:

- проект как центр;
- теория только под проблемы проекта;
- журнал ошибок;
- weekly demo;
- итоговый README.

Риск: много практики без осмысления.

### Existing course curriculum

Фокус на готовом курсе.

Структура:

- взять курс за основу;
- добавить свои проверки;
- заменить пассивные лекции задачами;
- каждую неделю делать артефакт.

Риск: пройти курс и не уметь применять.

### Live learning curriculum

Фокус на живом обучении:

- наставник;
- группа;
- публичный дневник;
- обсуждения;
- code review;
- воркшопи;
- митапи.

Риск: общение вместо практики.

## 10. Как не превратить Personal Curriculum в прокрастинацию

Красный флаг:

```text
Я уже неделю делаю план, но не сделал ни одной проверяемой попытки.
```

Правила:

1. Источников на старт: максимум 3-5.
2. Первый артефакт - в первые 7 дней.
3. Каждая неделя заканчивается проверкой.
4. План можно менять, но только после фактов.
5. Список "когда-нибудь" отдельно от текущего фокуса.
6. Нельзя добавлять новый источник без вопроса: что он закрывает?

# Часть II. Личный учебный план

## 11. Зачем нужен учебный план

Личный учебный план снимает ежедневную нагрузку выбора.

Без плана каждый день приходится решать:

- что учить;
- где открыть материал;
- сколько заниматься;
- что делать после;
- как проверить;
- когда повторить.

Это быстро утомляет. Хороший план принимает часть решений заранее.

Формула:

```text
один раз спроектировал -> много раз выполняешь -> раз в неделю корректируешь
```

## 12. Из чего состоит учебный план

| Элемент | Вопрос |
| --- | --- |
| Цель | Зачем я учусь? |
| Компетенции | Что должен уметь делать? |
| Источники | Откуда беру материал? |
| Расписание | Когда занимаюсь? |
| Задания | Что делаю руками? |
| Проверки | Как узнаю, что понял? |
| Feedback | Кто или что проверяет? |
| Повторение | Когда вернусь? |
| Артефакты | Что останется после блока? |
| Ревизия | Когда менять план? |

## 13. План от готового syllabus

Если есть университетский syllabus или хороший курс:

1. Возьми структуру как основу.
2. Перенеси недели и темы.
3. Добавь учебники и лекции.
4. Добавь задания, если их нет.
5. Добавь самопроверки.
6. Добавь итоговый артефакт.
7. Удали лишнее под свою цель.

Важно: чужой syllabus - не закон. Это сырье для личного плана.

## 14. План от карты компетенций

Если готового курса нет:

1. Определи роль или область.
2. Выпиши компетенции.
3. Раздели на базовые, рабочие, продвинутые.
4. Для каждой компетенции задай доказательство.
5. Найди источники только под текущий блок.
6. Собери план на 2-4 недели.

Пример:

```text
Не: "учить backend".
А:
1. написать HTTP handler;
2. проверить JSON response;
3. обработать error path;
4. написать integration test;
5. объяснить timeout и retry.
```

## 15. Один главный трек

Для серьезного самообучения лучше не брать 5 равных направлений.

Рабочая формула:

```text
один главный трек + один поддерживающий мини-трек + один легкий интерес без KPI
```

Пример:

| Роль | Пример |
| --- | --- |
| Главный трек | Go backend |
| Поддерживающий мини-трек | английский для документации |
| Легкий интерес | музыка, дикция, рисование |

Так снижается распыление, но сохраняется живость.

## 16. Что должно быть в недельном плане

Минимум:

```text
1. Главная компетенция недели.
2. 2-4 учебных блока.
3. 1 практическое задание.
4. 1 feedback-точка.
5. 1 повтор.
6. 1 недельный обзор.
```

Если неделя не содержит проверки, она легко становится неделей потребления информации.

# Часть III. Самообучение / самообразование

## 17. Что такое самообучение

Самообучение - это обучение, где внешний контроль слабый или отсутствует.

Ты сам отвечаешь за:

- выбор цели;
- выбор источников;
- расписание;
- практику;
- обратную связь;
- повторение;
- корректировку;
- завершение или отказ.

Это даёт свободу, но также создаёт риски:

- хаос;
- одиночество;
- отсутствие проверки;
- распыление;
- накопление источников;
- срыв расписания;
- иллюзия прогресса.

## 18. Главные ошибки самообучения

1. Бессистемность.
2. Слишком много направлений.
3. Отсутствие самопроверки.
4. Одна точка зрения вместо нескольких источников.
5. Нет графика.
6. Нет практики.
7. Нет обратной связи.
8. Нет критерия завершения.
9. План больше действия.
10. Продолжение неподходящего курса из-за жалости к вложенному времени.

## 19. Базовые правила самообучения

1. Выбери одну область на цикл.
2. Определи, зачем она нужна.
3. Найди базу, без которой нельзя двигаться дальше.
4. Возьми 1-3 главных источника.
5. Встрой учебу в календарь.
6. Делай задания.
7. Проверяй себя.
8. Получай feedback.
9. Фиксируй закрытые циклы.
10. Раз в неделю пересматривай план.

## 20. SOS-план для спада

Спад мотивации нормален. Его лучше готовить заранее.

Шаблон:

```text
Если я 3 раза подряд пропустил учебу:
1. Проверяю сон и нагрузку.
2. Уменьшаю блок до 15 минут.
3. Возвращаюсь к причине "зачем".
4. Делаю одну простую задачу.
5. Убираю один лишний источник.
6. Если отторжение сохраняется, делаю паузу и ревизию цели.
```

## 21. Право на отказ

Не всякое начатое обучение нужно завершать.

Отказ разумен, если:

- цель больше не актуальна;
- источник плохой;
- курс не ведёт к навыку;
- тема не нужна сейчас;
- обучение разрушает состояние;
- есть более важный трек;
- продолжение связано только с "жалко бросить".

Но перед отказом стоит провести короткую диагностику:

```text
Мне не подходит тема?
Мне не подходит источник?
Мне не подходит уровень?
Мне не подходит расписание?
Мне нужна пауза, а не отказ?
```

# Часть IV. Self-regulated learning / саморегулируемое обучение

## 22. Что такое SRL

Self-regulated learning / саморегулируемое обучение - это способность управлять собственным обучением.

SRL включает:

- постановку целей;
- выбор стратегии;
- мониторинг прогресса;
- замечание ошибок;
- оценку результата;
- изменение плана.

Это не техника запоминания. Это пульт управления учебной системой.

## 23. Цикл SRL

```text
планирую -> действую -> отслеживаю -> оцениваю -> корректирую
```

Или более подробно:

```text
цель -> стратегия -> попытка -> данные -> вывод -> новый план
```

## 24. Что отслеживать

Не надо отслеживать всё.

Достаточно 5 вопросов:

1. Что я пытался сделать?
2. Что получилось без подсказки?
3. Где была ошибка?
4. Какой feedback получил?
5. Что меняю в следующем блоке?

## 25. Метакогнитивные вопросы

Перед блоком:

```text
Что я хочу уметь после этого блока?
Как узнаю, что получилось?
Где могу ошибиться?
```

После блока:

```text
Что я реально могу сделать?
Что оказалось труднее?
Что требует повторения?
Что убрать или упростить?
```

## 26. Когда SRL становится бюрократией

SRL вреден, если:

- обзоры длиннее практики;
- ты больше заполняешь таблицы, чем учишься;
- система вызывает вину;
- нет решений "убрать";
- отслеживаются десятки метрик;
- план не меняется после фактов.

Правило:

```text
SRL должен уменьшать хаос, а не добавлять второй курс управления курсом.
```

# Часть V. Недельная калибровка обучения

## 27. Зачем нужна недельная калибровка

Учебная система склонна разрастаться.

Появляются:

- новые карточки;
- новые заметки;
- новые AI-workflow;
- новые курсы;
- новые "обязательные" ритуалы;
- новые списки источников;
- новые таблицы прогресса.

Недельная калибровка нужна, чтобы система не только добавляла, но и удаляла.

Главный вопрос:

```text
Что помогало закрывать учебные циклы, а что только создавало шум?
```

## 28. Мини-обзор на 20 минут

Раз в неделю:

```text
1. Что я изучал?
2. Что теперь могу сделать без подсказки?
3. Какие циклы закрыты?
4. Какие ошибки повторяются?
5. Что требует повторения?
6. Что требует практики?
7. Что убрать из системы?
8. Какая одна компетенция следующей недели?
```

## 29. Правило "оставить, изменить, убрать"

Каждую неделю выбери:

```text
Оставить: что реально помогает.
Изменить: что полезно, но слишком тяжело.
Убрать: что не ведёт к попытке, feedback или повтору.
```

Пример:

```text
Оставить: 3 строки exit ticket после блока.
Изменить: Anki не каждый день по часу, а 15 минут через день.
Убрать: бесконечный backlog YouTube-лекций.
```

## 30. Недельная калибровка в Obsidian

```markdown
# Weekly Learning Review

## Закрытые циклы
- 

## Доказательства навыка
- 

## Повторяющиеся ошибки
- 

## Что перегружает систему
- 

## Что убрать
- 

## Компетенция следующей недели
- 

## Первый учебный блок
- 
```

## 31. Ошибки недельной калибровки

1. Делать обзор на час и больше.
2. Считать количество часов вместо циклов.
3. Ругать себя вместо настройки системы.
4. Добавлять новые методы каждую неделю.
5. Не удалять лишнее.
6. Не выбирать один фокус.
7. Превращать review в дневник тревоги.

Главное правило:

```text
Недельная калибровка должна завершаться решением, а не самоотчётом.
```

# Часть VI. Расписание самообучения

## 32. Зачем расписание

В самообучении нет внешнего звонка, преподавателя и дедлайна.

Расписание заменяет внешний каркас.

Но расписание должно быть не героическим, а жизнеспособным.

Плохой план:

```text
с понедельника каждый день по 5 часов
```

Лучше:

```text
3 раза в неделю по 45-75 минут + 1 короткий повтор + weekly review
```

## 33. Аудит времени

Перед планированием 3-7 дней посмотри, куда реально уходит время.

Записывай:

- работа;
- дорога;
- бытовые дела;
- отдых;
- сон;
- соцсети;
- семейные дела;
- свободные окна;
- время высокой ясности;
- время низкой энергии.

Время не появляется из ниоткуда. Чтобы добавить обучение, что-то нужно убрать, сократить, делегировать или отложить.

Нельзя регулярно убирать сон и отдых.

## 34. Буферы

Люди часто недооценивают время.

Правила:

- добавляй 15-30 минут буфера к сложным блокам;
- не ставь учебу вплотную к тяжелым делам;
- не планируй максимум на каждый день;
- оставляй место для восстановления;
- не переносить весь недельный долг на выходные.

## 35. Форматы учебных блоков

### Блок 25 минут

Подходит для входа, повторения, короткой задачи.

```text
20 минут работа + 5 минут exit ticket
```

### Блок 45 минут

Подходит для чтения, конспекта, практики.

```text
35 минут работа + 5 минут проверка + 5 минут следующий шаг
```

### Блок 90 минут

Подходит для deep work и проекта.

```text
10 минут постановка задачи
60 минут практика
10 минут feedback/check
10 минут фиксация
```

## 36. Расписание по типам задач

| Тип задачи | Лучшее место |
| --- | --- |
| Новая сложная теория | окно ясности |
| Практика | стабильный защищенный слот |
| Повторение | короткие окна |
| Конспект | после чтения или лекции |
| Weekly review | один постоянный день |
| Легкое аудио | прогулка или дорога |
| Ревизия источников | отдельный короткий блок |

## 37. Стартовый ритуал

Ритуал нужен не для магии, а для снижения трения.

Примеры:

- открыть одну и ту же папку;
- убрать телефон;
- поставить таймер;
- налить воду;
- открыть первый файл;
- написать первый шаг;
- 2 минуты дыхания;
- 10 приседаний;
- короткая прогулка.

Правило:

```text
Ритуал должен вести к началу задачи, а не заменять задачу.
```

## 38. Пропуски

Пропуск не обнуляет обучение.

Нормальный протокол:

```text
1. Не компенсировать пропуск двойной нагрузкой автоматически.
2. Посмотреть причину.
3. Сделать минимальный возвратный блок.
4. Упростить следующий шаг.
5. Если пропуск повторяется, пересмотреть расписание.
```

# Часть VII. Учебные циклы

## 39. Главный учебный цикл

Минимальный цикл:

```text
прогноз -> попытка -> факт -> feedback -> повтор
```

Расшифровка:

- прогноз: что я ожидаю;
- попытка: что делаю сам;
- факт: что получилось;
- feedback: где расхождение;
- повтор: похожая задача с поправкой.

## 40. Почему цикл важнее материала

Материал дает вход.

Цикл создает обучение.

Пассивная схема:

```text
посмотрел -> понял -> пошел дальше
```

Рабочая схема:

```text
посмотрел -> попробовал -> ошибся -> исправил -> повторил
```

## 41. Учебный блок как цикл

Шаблон на 45 минут:

```text
5 минут - цель и прогноз
25 минут - самостоятельная попытка
10 минут - проверка / feedback
5 минут - exit ticket и следующий шаг
```

## 42. Цикл Колба

Модель Колба хорошо ложится на практические навыки:

```text
опыт -> рефлексия -> вывод -> новый эксперимент
```

Применение:

1. Сделал задачу.
2. Заметил, где сломалось.
3. Сформулировал принцип.
4. Проверил принцип в новой задаче.

## 43. Exit ticket

После блока дольше 25 минут запиши 3 строки:

```text
Главное:
Неясно:
Следующий проверяемый шаг:
```

Это маленький способ закрыть цикл и не входить в следующую сессию с нуля.

## 44. Ошибки учебных циклов

1. Пропускать самостоятельную попытку.
2. Просить AI решить задачу до своей попытки.
3. Получать feedback, но не делать повтор.
4. Фиксировать эмоцию вместо факта.
5. Делать цикл слишком бюрократичным.
6. Не выбирать следующий шаг.

# Часть VIII. Компетенция вместо темы

## 45. Почему тема слабее компетенции

Фраза:

```text
изучить Docker
```

слишком широкая.

Лучше:

```text
собрать Dockerfile для Go-сервиса, запустить контейнер, пробросить порт, передать env-переменные и объяснить слои образа
```

Тема создает направление.

Компетенция создает критерий проверки.

## 46. Формула компетенции

```text
Я могу [действие] в [условиях] с [критерием качества].
```

Примеры:

```text
Я могу написать unit test для handler, проверить success и error case.
```

```text
Я могу объяснить различие REST и message queue на примере заказа и оплаты.
```

```text
Я могу прочитать простую научную статью и выделить тезис, метод, данные и ограничение.
```

## 47. Критерии хорошей компетенции

Компетенция должна быть:

- наблюдаемой;
- проверяемой;
- достаточно маленькой;
- связанной с практикой;
- привязанной к контексту;
- не сводящейся к одному механическому действию;
- имеющей доказательство.

## 48. Как превращать темы в компетенции

| Тема | Компетенция |
| --- | --- |
| Go interfaces | написать функцию, зависящую от маленького интерфейса, и протестировать fake |
| SQL indexes | объяснить, когда индекс помогает, и проверить план запроса |
| Английская грамматика | написать 10 предложений с конструкцией и исправить ошибки |
| Когнитивная нагрузка | найти 3 источника перегруза в учебном материале и предложить исправления |
| Конспекты | превратить лекцию в вопросы для самопроверки |

## 49. Уровни владения компетенцией

| Уровень | Признак |
| --- | --- |
| 0 | Не могу объяснить |
| 1 | Узнаю термин |
| 2 | Могу объяснить словами |
| 3 | Могу решить типовую задачу |
| 4 | Могу перенести в новую задачу |
| 5 | Могу объяснить другому и защитить решение |

## 50. Доказательства компетенции

Доказательством может быть:

- задача;
- тест;
- проект;
- объяснение;
- публичная заметка;
- code review;
- исправленная ошибка;
- портфолио;
- сравнительная таблица;
- дизайн-документ;
- запись выступления;
- повторная похожая задача.

Не доказательство:

- "я читал";
- "я смотрел курс";
- "я вроде понял";
- "у меня есть конспект";
- "AI объяснил, и мне стало ясно".

# Часть IX. Competency-Based Education / CBE

## 51. Что такое CBE

Competency-Based Education / CBE - подход, где прогресс определяется не количеством часов, а доказанным владением компетенциями.

Обычная модель:

```text
прошел курс -> получил оценку -> двигаешься дальше
```

CBE:

```text
освоил компетенцию -> подтвердил доказательством -> двигаешься дальше
```

Для самообучения это особенно полезно.

## 52. CBE для личной системы

Личная CBE-система:

```text
карта компетенций -> критерии -> задания -> доказательства -> feedback -> повторная проверка
```

Шаблон:

```markdown
# Компетенция

## Что должен уметь делать?

## Критерии освоения

## Проверочное задание

## Доказательство

## Типичные ошибки

## Feedback

## Повторная попытка

## Следующий уровень
```

## 53. Prior Learning Assessment / оценка уже имеющихся навыков

Не всегда нужно начинать с нуля.

Сначала можно провести инвентаризацию:

```text
Что я уже умею без подсказки?
Какие есть доказательства?
Где я только узнаю термин, но не применяю?
Какие ошибки повторяются?
Какой минимальный тест покажет владение?
```

Это помогает не тратить месяц на то, что уже частично освоено.

Но есть риск самообмана. Нужны доказательства.

## 54. CBE и портфолио

Для практических областей полезно собирать портфолио доказательств:

- репозиторий;
- README;
- тесты;
- заметки по ошибкам;
- публичные статьи;
- архитектурные схемы;
- мини-проекты;
- разборы кейсов;
- демонстрационные видео.

Портфолио не должно быть только витриной. Оно должно показывать рост компетенций.

## 55. Ошибки CBE

1. Формализовать каждое микродействие.
2. Путать компетенцию с темой.
3. Отмечать "освоено" без доказательства.
4. Делать чеклисты вместо практики.
5. Мерить только типовые задачи и забывать перенос.
6. Игнорировать сложные навыки, где нужны проекты и наблюдение.

# Часть X. Mastery learning

## 56. Что такое mastery learning

Mastery learning - это логика обучения, где человек двигается дальше после достижения заданного уровня владения, а не просто после истечения времени.

Формула:

```text
объяснение -> практика -> проверка -> коррекция -> повторная проверка -> следующий блок
```

Главный принцип:

```text
Не двигаться дальше с критическими пробелами в базе.
```

## 57. Чем mastery learning отличается от обычного курса

Обычный курс:

```text
неделя прошла -> следующая тема
```

Mastery learning:

```text
критерий освоения выполнен -> следующая тема
```

Это особенно важно в темах, где новое строится на старом:

- математика;
- языки;
- программирование;
- музыка;
- медицина;
- инженерные навыки;
- базы данных;
- системное мышление.

## 58. Мини-критерий mastery

Для самообучения не нужно ждать абсолютного совершенства.

Достаточно критерия:

```text
Я могу выполнить базовую задачу без подсказки, объяснить решение, исправить типичную ошибку и повторить похожую задачу через время.
```

## 59. Уровни mastery для темы

| Уровень | Действие |
| --- | --- |
| База | знаю термины и могу объяснить |
| Типовая задача | выполняю по образцу |
| Самостоятельность | выполняю без подсказки |
| Ошибки | нахожу и исправляю типичные сбои |
| Перенос | применяю в новой ситуации |
| Объяснение | могу научить другого |

## 60. Риск mastery learning

Риск - застрять и не двигаться дальше, потому что "ещё не идеально".

Правило:

```text
Mastery - не перфекционизм.
Mastery - достаточное владение для следующего шага.
```

# Часть XI. Learning in Public

## 61. Что такое Learning in Public

Learning in Public - публичное обучение: человек делится процессом, выводами, ошибками, проектами и вопросами.

Форматы:

- блог;
- Telegram-канал;
- GitHub;
- Хабр;
- YouTube;
- заметки;
- стримы;
- open source;
- доклады;
- туториалы;
- публичный журнал ошибок;
- еженедельные отчёты.

## 62. Зачем учиться публично

Польза:

- структурировать знания;
- получать feedback;
- находить единомышленников;
- уменьшать одиночество;
- строить портфолио;
- учиться объяснять;
- видеть прогресс;
- помогать новичкам;
- находить наставников.

Learning in Public хорошо сочетается с CBE:

```text
каждая публичная заметка или проект может быть доказательством компетенции
```

## 63. Как начать безопасно

1. Выбери одну платформу.
2. Не обещай экспертность.
3. Пиши как ученик: что попробовал, где ошибся, что понял.
4. Не публикуй конфиденциальное.
5. Не гонись за лайками.
6. Делай маленькие регулярные публикации.
7. Отделяй учебный дневник от личной исповеди.
8. Проси конкретную обратную связь.

## 64. Форматы Learning in Public

### Учебный лог

```text
Сегодня изучал:
Что сделал:
Где ошибся:
Что понял:
Следующий шаг:
```

### Ошибка недели

```text
Ошибка:
Почему возникла:
Как нашел:
Как исправил:
Что теперь проверяю:
```

### Мини-туториал

```text
Проблема -> решение -> пример -> типичная ошибка -> проверка
```

### Публичный проект

```text
README -> задачи -> коммиты -> выводы -> демо
```

### Weekly build in public

```text
что строю -> что сделал -> что не получилось -> что делаю дальше
```

## 65. Ошибки Learning in Public

1. Ориентироваться на просмотры.
2. Публиковать больше, чем учиться.
3. Делать вид эксперта, когда ты ученик.
4. Раскрывать лишние личные или рабочие данные.
5. Спорить вместо уточнения.
6. Бояться публиковать неидеальное.
7. Делать контент вместо артефактов.

Главное правило:

```text
Публичность должна усиливать обучение, а не заменять его.
```

# Часть XII. Lifelong learning

## 66. Что такое lifelong learning

Lifelong learning - непрерывное обучение в течение жизни.

В рабочем смысле это не романтическая идея "учиться всему всегда".

Это система:

```text
замечать устаревание навыков -> выбирать важный пробел -> строить короткий цикл практики -> получать feedback -> фиксировать результат -> обновлять траекторию
```

## 67. Зачем учиться всю жизнь

1. Профессии меняются.
2. Инструменты устаревают.
3. Рынок требует переобучения.
4. Мозгу полезна когнитивная активность.
5. Новые навыки расширяют свободу действий.
6. Обучение поддерживает чувство компетентности.
7. Смена траектории становится менее пугающей.

Но не всякое обучение полезно.

Пассивное накопление курсов - слабая версия lifelong learning.

## 68. Четыре корзины lifelong learning

| Корзина | Примеры |
| --- | --- |
| Профессия | Go, backend, DevOps, system design |
| Инструменты | AI, Obsidian, Anki, автоматизация |
| Мышление | логика, критическое мышление, архитектура |
| Состояние | сон, стресс-регуляция, физическая активность |

У каждой корзины может быть свой ритм.

Не нужно одновременно прокачивать всё с одинаковой интенсивностью.

## 69. Режим lifelong learning

Практичная формула:

```text
каждый квартал - один главный учебный сезон;
каждую неделю - одна компетенция;
каждый блок - один закрытый цикл;
каждый месяц - ревизия траектории.
```

## 70. Фильтр для новых тем

Перед добавлением темы спроси:

```text
1. Это нужно мне сейчас?
2. Это связано с текущей траекторией?
3. Можно применить в ближайшие 7-14 дней?
4. Это закрывает реальный пробел?
5. У меня есть время на практику?
```

Если нет - тема идет в backlog, а не в текущий план.

# Часть XIII. Mindshift / смена учебной траектории

## 71. Что такое Mindshift

Mindshift - смена учебной или профессиональной траектории во взрослом возрасте.

Главная идея:

```text
текущая идентичность не обязана быть окончательной
```

Но смена траектории не происходит одной мотивацией.

Нужны:

- реалистичный срок;
- пробные проекты;
- практика;
- обратная связь;
- база;
- среда;
- финансовая и эмоциональная устойчивость;
- готовность быть новичком.

## 72. Проверяемая гипотеза вместо мечты

Не так:

```text
Я стану backend-разработчиком.
```

Лучше:

```text
В течение 4 недель я проверю, могу ли регулярно решать backend-задачи, читать документацию, писать маленький сервис и получать feedback.
```

Смена траектории начинается с проверки реальности.

## 73. Протокол смены направления

1. Назови новую область.
2. Объясни, зачем она тебе.
3. Найди 3 реальные задачи из области.
4. Сделай одну пробную задачу.
5. Посмотри, какая база нужна.
6. Найди людей, которые уже прошли путь.
7. Собери учебный цикл на 2-4 недели.
8. Оцени результат без драматизации.

## 74. Что важно не преувеличить

Слабая формулировка:

```text
Можно стать кем угодно, если захотеть.
```

Более точная:

```text
Взрослые могут осваивать новые навыки и менять траекторию, но результат зависит от времени, здоровья, среды, стартовой базы, практики, обратной связи и требований области.
```

## 75. Mindshift и право быть новичком

При смене траектории человек часто теряет прежний статус.

Нужно заранее принять:

- сначала будет медленно;
- ошибки неизбежны;
- сравнение с профессионалами вредно;
- старые навыки могут пригодиться иначе;
- не всё нужно бросать ради нового;
- первые проекты будут простыми.

# Часть XIV. Практические сценарии

## 76. Сценарий: хочу начать самообучение с нуля

1. Выбери одну область.
2. Сформулируй зачем.
3. Сделай K-W-L или карту начальных знаний.
4. Найди 1 основной источник и 1 дополнительный.
5. Определи первую компетенцию.
6. Запланируй 2-3 блока в неделю.
7. Сделай первый артефакт в первые 7 дней.

## 77. Сценарий: много источников и хаос

1. Останови добавление новых материалов на неделю.
2. Составь список текущих источников.
3. Оставь максимум 3.
4. Для каждого источника напиши, какую компетенцию он закрывает.
5. Остальное перенеси в backlog.
6. Сделай одну практическую задачу.

## 78. Сценарий: курс прохожу, но навыка нет

1. Выпиши темы курса.
2. Переведи каждую тему в компетенцию.
3. Добавь проверочное задание.
4. Сделай самостоятельную попытку.
5. Получи feedback.
6. Сделай повторную задачу.
7. Только потом двигайся дальше.

## 79. Сценарий: самообучение зашло в тупик

1. Признай факт тупика.
2. Проверь усталость и сон.
3. Сделай информационный вакуум на 2-7 дней, если есть перегруз.
4. Вернись к исходному "зачем".
5. Проведи аудит знаний.
6. Найди пробел или новую форму применения.
7. Уменьши план.
8. Если цель больше не нужна, завершай направление.

## 80. Сценарий: нужно составить учебную программу на месяц

1. Тема месяца.
2. Не-цели месяца.
3. 3-5 источников.
4. 4 недельных фокуса.
5. Компетенция каждой недели.
6. Проверка каждой недели.
7. Итоговый артефакт.
8. Дата финального обзора.

## 81. Сценарий: хочу учиться публично

1. Выбери платформу.
2. Определи границы публичности.
3. Публикуй раз в неделю.
4. Формат: что сделал, где ошибся, что понял.
5. Не обещай экспертность.
6. Собирай feedback.
7. Переноси полезное в план.

## 82. Сценарий: меняю профессию или траекторию

1. Опиши новую роль.
2. Собери карту компетенций.
3. Найди 3 людей или 3 портфолио в этой роли.
4. Сделай пробный проект.
5. Проведи PLA-инвентаризацию.
6. Построй 4-недельный цикл.
7. Через месяц реши: продолжать, изменить, отложить.

# Часть XV. Фишки и техники

## 83. Фишки проектирования учебной системы

1. Начинай не с источников, а с результата.
2. На один цикл выбирай одну главную компетенцию.
3. Источник без задания - это только входной материал.
4. Каждая неделя должна иметь артефакт.
5. Пиши "не-цели", чтобы не расползаться.
6. Делай backlog отдельно от текущего плана.
7. Не добавляй новый метод без удаления старого.
8. Если система не помогает начать задачу, упрости её.
9. План должен помещаться на одну страницу.
10. Первый артефакт должен появиться быстро, не в конце года.

## 84. Фишки расписания

1. Сначала аудит времени, потом план.
2. Не жертвуй сном ради регулярности.
3. Добавляй буфер к учебному блоку.
4. Не ставь сложное на время низкой энергии.
5. Не переносить весь долг на воскресенье.
6. Планируй маленький возврат после пропуска.
7. Используй постоянный слот для weekly review.
8. Делай первый шаг заранее.
9. Ставь короткий повтор в отдельное окно.
10. Оставляй место для жизни, иначе план не выживет.

## 85. Фишки компетенций

1. Любую тему спрашивай: "что я должен уметь?"
2. Добавляй критерий "без подсказки".
3. Добавляй критерий "в новой ситуации".
4. Не ставь "освоено" без доказательства.
5. Для сложного навыка делай 3 уровня: база, практика, перенос.
6. Компетенция должна быть меньше месяца, но больше одного клика.
7. Ошибка показывает слабую компетенцию.
8. Если компетенцию нельзя проверить, переформулируй.
9. Если компетенция слишком широкая, раздроби.
10. Если слишком мелкая, свяжи с контекстом.

## 86. Фишки учебных циклов

1. Перед задачей записывай прогноз.
2. Сначала попытка, потом подсказка.
3. AI подключай после своей попытки.
4. Feedback должен вести к следующему действию.
5. Ошибку записывай как расхождение модели с фактом.
6. После feedback делай похожую задачу.
7. Не закрывай блок без exit ticket.
8. Один цикл лучше трех часов пассивного просмотра.
9. Если цикл занимает много подготовки, сократи шаблон.
10. Видимый прогресс - это закрытый цикл.

## 87. Фишки недельной калибровки

1. Review должен занимать 20-30 минут.
2. Первый вопрос: что теперь могу сделать?
3. Второй вопрос: что повторяется как ошибка?
4. Третий вопрос: что убрать?
5. Выбирай одну компетенцию следующей недели.
6. Не добавляй 5 экспериментов сразу.
7. Не оценивай себя как личность, оценивай систему.
8. Смотри на доказательства, а не настроение.
9. Если прогресса не видно, меняй метрики.
10. Если всё перегружено, возвращайся к минимальному циклу.

## 88. Фишки Learning in Public

1. Публикуй не "я эксперт", а "я проверил".
2. Пиши маленькие заметки.
3. Самая полезная публикация - разбор ошибки.
4. Не публикуй конфиденциальное.
5. Не измеряй обучение лайками.
6. Проси конкретный feedback.
7. Делай итог недели.
8. Публичный README проекта - сильный артефакт.
9. Учебный лог может стать портфолио.
10. Если публичность мешает учиться, уменьши её.

## 89. Фишки lifelong learning и Mindshift

1. Не учи всё новое сразу.
2. Делай квартальные учебные сезоны.
3. Раз в месяц проверяй, не устарела ли цель.
4. Смена траектории начинается с маленького проекта.
5. Не делай вывод "я не способен" по первой трудности.
6. Сравнивай себя с прошлой версией.
7. Старые навыки могут стать преимуществом в новой области.
8. Умеренная новизна полезна, постоянный стресс - нет.
9. Backlog нужен, чтобы не тащить все интересы в текущую неделю.
10. Учиться всю жизнь - не значит всю жизнь смотреть курсы.

# Часть XVIII. 7-дневный эксперимент

## 104. Эксперимент: учебная архитектура без перегруза

Цель: проверить, помогает ли минимальная система чаще закрывать учебные циклы.

Правила на 7 дней:

1. Одна главная компетенция.
2. Максимум 3 источника.
3. Минимум 3 учебных блока.
4. Каждый блок закрывается exit ticket.
5. Один feedback.
6. Одна повторная похожая задача.
7. В конце недели - калибровка.

## 105. Метрики

| Метрика | Вопрос |
| --- | --- |
| Старт | Начал ли я без долгой подготовки? |
| Попытка | Была ли самостоятельная попытка? |
| Feedback | Получил ли я корректирующий сигнал? |
| Повтор | Была ли повторная похожая задача? |
| Артефакт | Осталось ли доказательство? |
| Фокус | Снизилось ли распыление? |
| Перегруз | Система помогала или мешала? |

## 106. Решение после 7 дней

Оставить систему, если:

- стало яснее, что делать;
- появились закрытые циклы;
- меньше хаоса в источниках;
- виден конкретный прогресс;
- недельный обзор дал решение.

Упростить, если:

- слишком много полей;
- много планирования;
- мало практики;
- feedback запаздывает.

Убрать часть системы, если:

- она создаёт вину;
- занимает больше времени, чем обучение;
- не ведёт к попытке;
- не помогает исправлять ошибки.

# Часть XIX. Чего не стоит делать

## 107. Главные ошибки

1. Делать учебный план без заданий.
2. Собирать 30 источников перед первой попыткой.
3. Учить 5 направлений как одинаково главные.
4. Измерять прогресс часами.
5. Не формулировать компетенции.
6. Не иметь критериев освоения.
7. Двигаться дальше с критическими пробелами.
8. Ждать мотивации вместо расписания.
9. Жертвовать сном ради учебы.
10. Делать review без решения "что убрать".
11. Считать публичность заменой практики.
12. Считать lifelong learning оправданием бесконечного потребления.
13. Просить AI сделать работу до собственной попытки.
14. Не иметь права на паузу или отказ.
15. Путать смену траектории с импульсивным бегством от трудности.

## 108. Красные флаги системы

Система, вероятно, не работает, если:

- план растёт, а артефактов нет;
- источников больше, чем задач;
- "изучал" чаще, чем "сделал";
- weekly review вызывает только вину;
- нет повторных попыток;
- нет feedback;
- ты не можешь назвать текущую компетенцию;
- обучение постоянно переносится на идеальное будущее;
- всё держится на мотивации;
- нет критерия завершения.

# Часть XX. Финальная модель

## 109. Минимальная учебная ОС

Если убрать всё лишнее, нужна такая система:

```text
1. Backlog тем.
2. Текущий учебный сезон.
3. Карта компетенций.
4. Расписание блоков.
5. Журнал учебных циклов.
6. Доказательства навыка.
7. Недельная калибровка.
```

## 110. Один учебный блок

```text
компетенция -> прогноз -> попытка -> feedback -> повтор -> exit ticket
```

## 111. Одна учебная неделя

```text
одна компетенция -> 2-4 блока -> один feedback -> один артефакт -> review
```

## 112. Один учебный сезон

```text
тема -> карта -> задания -> доказательства -> итоговый артефакт -> решение о следующем сезоне
```

## 113. Самое короткое правило

```text
Учебная архитектура хороша, если она чаще приводит к самостоятельной попытке, обратной связи и повторной проверке.
```

Всё, что не помогает этому, должно быть упрощено, отложено или удалено.

---

## 115. Ограничения

Эта инструкция не утверждает, что любой личный учебный план автоматически работает.

Более точная позиция:

```text
Учебная архитектура полезна, если она снижает хаос и увеличивает число проверяемых самостоятельных попыток с feedback и повтором.
```

Сильнее всего локальные источники поддерживают отдельные компоненты:

- активное воспроизведение;
- обратная связь;
- практика;
- саморегуляция;
- видимый прогресс;
- работа с ошибками;
- распределение нагрузки;
- проверяемые компетенции.

Слабее доказаны конкретные личные шаблоны, названия трендов и универсальные формулы. Их нужно проверять на себе короткими экспериментами.
