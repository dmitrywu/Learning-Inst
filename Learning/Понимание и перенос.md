# Понимание и перенос - инструкция пользователя

Режим: `ytb`  
Тема: понимание, перенос знаний, interleaving, worked examples, fading, self-explanation, learning by teaching, generation effect, pre-testing, transfer practice, сенсорная обратная связь.  
Основные источники: `Learning`, `Memory`, `4sort`.

---

## 1. Главная идея

Понимание - это не ощущение "мне ясно, пока я смотрю на объяснение". Понимание начинается там, где человек может:

- объяснить идею своими словами;
- показать простой пример;
- назвать антипример;
- отличить от похожей идеи;
- применить в новой задаче;
- найти типичную ошибку;
- исправить свою модель после feedback.

Перенос - это следующий уровень. Это способность использовать знание не только в знакомом упражнении, но и в новой ситуации.

Короткая формула:

```text
понял в примере -> объяснил шаги -> убрал подсказки -> сделал сам -> изменил контекст -> получил feedback -> повторил иначе
```

Если совсем кратко:

```text
Понимание = могу объяснить почему.
Перенос = могу применить не там, где учили.
```

## 2. Карта технологий

| Технология | Главный вопрос | Для чего нужна |
|---|---|---|
| Interleaving / чередование | Могу ли я выбрать правильный метод среди похожих? | Различение типов задач и перенос |
| Worked examples / разобранные примеры | Вижу ли я структуру правильного решения? | Снижение перегруза у новичка |
| Fading | Могу ли я решить, когда подсказки постепенно исчезают? | Переход от примера к самостоятельности |
| Self-explanation / самообъяснение | Почему этот шаг работает? | Связать шаги, принципы и ошибки |
| Learning by teaching | Могу ли я объяснить другому? | Организация материала и выявление пробелов |
| Generation effect | Могу ли я сам сформулировать ответ, пример или правило? | Активная переработка вместо пассивного чтения |
| Pre-testing | Что я уже могу предположить до изучения? | Усилить внимание к ответам и пробелам |
| Transfer practice | Могу ли я применить принцип в новой задаче? | Проверка настоящего понимания |
| Сенсорная обратная связь | Что показывает запись, звук, образец или телесное ощущение? | Коррекция речи, произношения, движения и навыков |

Эти методы не конкурируют. Они закрывают разные этапы:

```text
pre-testing -> worked example -> self-explanation -> fading -> generation -> transfer practice -> interleaving -> feedback
```

## 3. Что важно не преувеличивать

### Хорошо подтверждается

- Активная переработка материала обычно полезнее пассивного просмотра.
- Разобранные примеры помогают новичкам в сложных процедурных навыках.
- Постепенное снятие подсказок помогает перейти к самостоятельной задаче.
- Самообъяснение и объяснение другому могут улучшать понимание и организацию материала.
- Предварительные вопросы могут усилить внимание к последующему материалу.
- Interleaving часто полезен для различения похожих типов задач.
- Быстрая и точная обратная связь важна для обучения.

### Требует осторожности

- Не любая трудность полезна.
- Не всякое перемешивание является interleaving.
- Не любой пример учит: пассивный просмотр готовых решений может дать иллюзию понимания.
- Не всякая подсказка полезна: избыток подсказок перегружает.
- Не любое объяснение глубоко: можно пересказать текст и не понять.
- Pre-testing без последующего изучения и feedback может закрепить путаницу.
- Сенсорная обратная связь не заменяет осмысленную коррекцию.

Правильный фильтр:

```text
помогает ли метод сделать следующую проверяемую попытку?
```

## 4. Базовый цикл понимания и переноса

### Шаг 1. Сделать pre-test

До изучения:

```text
Что я думаю об этой теме?
Какие вопросы у меня есть?
Как бы я решил задачу сейчас?
Где я не уверен?
```

Цель - не угадать правильно, а включить внимание и создать крючки для будущего ответа.

### Шаг 2. Разобрать пример

Не начинать с хаотичной самостоятельной борьбы, если тема новая и сложная.

Хороший worked example показывает:

- задачу;
- цель;
- ограничения;
- шаги;
- почему шаги такие;
- где можно ошибиться;
- как проверить результат.

### Шаг 3. Объяснить шаги

После примера:

```text
что происходит?
почему этот шаг работает?
какой принцип здесь используется?
где типичная ошибка?
```

### Шаг 4. Убрать часть подсказок

Fading:

```text
полный пример -> пример с пропущенным шагом -> задача с намеком -> самостоятельная задача
```

### Шаг 5. Сгенерировать свой вариант

Generation effect:

- свой пример;
- свое правило;
- свой вопрос;
- свой антипример;
- свое решение;
- свой тест.

### Шаг 6. Сделать transfer practice

Изменить контекст:

- другая формулировка;
- другие данные;
- другой порядок;
- другой интерфейс;
- новый кейс;
- похожая, но не та же задача;
- реальный проект.

### Шаг 7. Перемешать похожие типы задач

Interleaving:

```text
A A A -> B B B -> C C C
```

заменить на:

```text
A -> B -> A -> C -> B -> C -> A
```

Но только после минимального понимания каждого элемента.

### Шаг 8. Получить feedback

Feedback должен быть:

- быстрым;
- конкретным;
- связанным с ошибкой;
- не слишком объемным;
- ведущим к следующей попытке.

## 5. Interleaving / чередование

### 5.1. Что это

Interleaving - это чередование похожих типов задач, тем или случаев, чтобы учиться выбирать правильный метод.

Не так:

```text
10 задач только на один тип
10 задач только на второй тип
10 задач только на третий тип
```

А так:

```text
тип A -> тип B -> тип A -> тип C -> тип B -> тип C
```

Главная польза не в разнообразии ради развлечения. Польза в том, что мозг вынужден распознавать тип ситуации.

### 5.2. Когда применять

Interleaving особенно полезен, когда:

- темы похожи и путаются;
- нужно выбрать метод, а не просто выполнить известную процедуру;
- задача на экзамене или работе не будет подписана названием темы;
- надо готовиться к реальным кейсам;
- есть иллюзия "я понял", потому что все упражнения были однотипными.

Примеры:

- математика: разные типы задач на выбор метода;
- программирование: slice, map, pointer receiver, interface, error handling;
- язык: похожие времена, предлоги, конструкции;
- право: разные основания увольнения или виды доказательств;
- обучение: retrieval, spacing, interleaving, elaboration;
- медицина: похожие симптомы и дифференциальные признаки.

### 5.3. Когда не применять сразу

Новичку часто нужна короткая блоковая фаза:

```text
сначала понять A;
потом понять B;
потом сравнить A и B;
потом перемешивать.
```

Если смешать всё до базового понимания, получится не продуктивная трудность, а перегруз.

### 5.4. Протокол interleaving

1. Выберите 2-4 похожие темы.
2. Для каждой темы сформулируйте отличительный признак.
3. Сделайте 2-3 простые задачи отдельно.
4. Сравните темы: чем похожи, чем отличаются.
5. Смешайте задачи.
6. Перед решением каждой задачи сначала назовите тип.
7. После ошибки запишите, с чем спутали.

Шаблон:

```text
Задача:
Какой это тип?
По какому признаку я это понял?
Какой метод нужен?
С чем можно спутать?
Что показал feedback?
```

### 5.5. Фишки interleaving

- Перемешивайте близкие, но различимые темы.
- Перед решением называйте тип задачи.
- Делайте карточки не только "что это", но и "чем отличается".
- Если путаетесь полностью, вернитесь к worked example.
- Если все легко, добавьте новый контекст.
- В Anki не дробите всё на слишком узкие колоды: смешанные повторы часто полезнее.
- В weekly review выбирайте 2-3 пары тем, которые нужно различить.

### 5.6. Ошибки interleaving

Ошибка: смешивать случайные далекие темы.

Плохо:

```text
Go interfaces -> история Японии -> трудовое право -> дыхательные практики
```

Это может быть режим дня, но не сильный interleaving.

Лучше:

```text
Go interface -> method set -> fake -> mock -> pointer receiver
```

Ошибка: путать сложность с пользой.

Если вы не можете даже начать, нужна поддержка:

```text
пример -> объяснение -> частичная подсказка -> потом смешивание
```

## 6. Worked examples / разобранные примеры

### 6.1. Что это

Worked example - это готовое решение с объяснением шагов.

Он полезен, когда:

- тема новая;
- задача процедурная;
- много взаимосвязанных элементов;
- новичок не знает, с чего начать;
- самостоятельная попытка превращается в угадывание.

Смысл worked example:

```text
сначала увидеть структуру решения,
потом объяснить её,
потом повторить,
потом решить самому.
```

### 6.2. Что должно быть в хорошем примере

Хороший пример содержит:

- условие задачи;
- цель;
- исходные ограничения;
- шаги решения;
- объяснение каждого шага;
- проверку результата;
- типичную ошибку;
- близкую задачу для самостоятельной попытки.

Плохой пример:

- слишком большой;
- без объяснения;
- с магическими переходами;
- без проверки;
- без последующей задачи;
- не соответствует уровню ученика.

### 6.3. Протокол worked example

```text
1. Прочитать условие.
2. Предсказать первый шаг.
3. Посмотреть шаг решения.
4. Объяснить, почему он сделан.
5. Повторить для каждого шага.
6. Закрыть пример и восстановить план.
7. Сделать близкую задачу.
```

Важно: пример не должен быть конечной точкой. Он должен вести к попытке.

### 6.4. Фишки worked examples

- Перед просмотром решения сделайте прогноз.
- После каждого шага спрашивайте "почему именно так?".
- Закрывайте пример и восстанавливайте план.
- Делайте задачу-близнец без подсказки.
- Сравнивайте свое решение с эталоном.
- Если пример слишком сложен, уменьшите задачу.
- Если пример слишком легок, переходите к fading или самостоятельной задаче.

### 6.5. Ошибки worked examples

Ошибка: смотреть примеры как видеоразвлечение.

Исправление:

```text
после примера всегда сделать прогноз, объяснение или задачу.
```

Ошибка: собирать много примеров перед первой попыткой.

Исправление:

```text
один пример -> одна близкая задача.
```

Ошибка: использовать готовое решение для списывания.

Исправление:

```text
сначала восстановить ход решения без источника.
```

## 7. Fading / постепенное снятие подсказок

### 7.1. Что это

Fading - это постепенный переход от полной поддержки к самостоятельному решению.

Цепочка:

```text
полный пример
-> пример с пропущенным последним шагом
-> пример с несколькими пропусками
-> задача с намеком
-> самостоятельная задача
```

Главная задача - не бросить новичка в воду слишком рано и не держать его на подсказках слишком долго.

### 7.2. Когда использовать

Использовать после worked example, если:

- вы понимаете пример, но не можете решить самостоятельно;
- самостоятельная задача слишком резкая;
- есть страх чистого листа;
- много ошибок из-за перегруза;
- нужно перейти от узнавания решения к воспроизведению.

### 7.3. Протокол fading

Уровень 1:

```text
решение полностью показано;
ваша задача - объяснить каждый шаг.
```

Уровень 2:

```text
показаны первые шаги;
вы дописываете последний.
```

Уровень 3:

```text
показана структура;
вы заполняете ключевые шаги.
```

Уровень 4:

```text
есть только условие и один намек.
```

Уровень 5:

```text
полностью самостоятельная задача.
```

### 7.4. Фишки fading

- Убирать не все подсказки сразу, а одну категорию.
- Сначала убрать ответы, потом намеки, потом структуру.
- Если ошибка повторяется, вернуть один уровень поддержки.
- Если стало легко, быстрее перейти к самостоятельной задаче.
- В программировании: сначала трассировать готовый код, потом дописать функцию, потом написать с нуля.
- В языке: сначала повторить образец, потом заменить часть фразы, потом сказать свою.

### 7.5. Ошибки fading

Ошибка: подсказки остаются навсегда.

Признак:

```text
без шаблона я ничего не могу начать.
```

Исправление:

```text
запланировать, какую подсказку убрать в следующей задаче.
```

Ошибка: подсказки убраны слишком рано.

Признак:

```text
я не понимаю, что делать, и угадываю.
```

Исправление:

```text
вернуться к примеру или частично заполненной задаче.
```

## 8. Self-explanation / самообъяснение

### 8.1. Что это

Self-explanation - это объяснение самому себе:

- что происходит;
- почему шаг работает;
- какой принцип применяется;
- где можно ошибиться;
- как проверить результат.

Это не длинный пересказ. Это короткая проверка понимания.

### 8.2. Базовые вопросы

После примера:

```text
Что здесь происходит?
Почему этот шаг нужен?
Какой принцип используется?
Что было бы, если шаг пропустить?
Где типичная ошибка?
Как проверить, что решение верное?
```

После ошибки:

```text
Что я ожидал?
Что произошло?
Где сломалась моя модель?
Какое правило я не заметил?
Какую похожую задачу решить?
```

### 8.3. Когда использовать

- после worked example;
- после ошибки;
- после feedback;
- после объяснения ИИ;
- после чтения сложного абзаца;
- перед созданием карточки;
- перед переходом к самостоятельной задаче.

### 8.4. Фишки self-explanation

- Ограничьте объяснение 3-7 предложениями.
- Обязательно добавляйте "где ошибка вероятна".
- Проверяйте объяснение по источнику или задаче.
- Если объяснение длинное и туманное, сократите до схемы.
- Если объяснение звучит красиво, но не ведет к действию, добавьте мини-задачу.
- Используйте формат "почему не другой способ?".

### 8.5. Ошибки self-explanation

Ошибка: убедительно объяснять себе неверную модель.

Исправление:

```text
self-explanation -> feedback -> повторная задача.
```

Ошибка: писать длинные рассуждения вместо проверки.

Исправление:

```text
3 вопроса: что, почему, где ошибка.
```

## 9. Learning by teaching / обучение через объяснение

### 9.1. Что это

Learning by teaching - обучение через объяснение другому человеку, воображаемому ученику или самому себе в роли преподавателя.

Сильная версия:

```text
изучил -> выделил структуру -> объяснил простыми словами -> привел пример -> назвал ошибку -> получил вопрос -> проверил
```

Слабая версия:

```text
пересказал текст почти теми же словами.
```

### 9.2. Почему работает

Когда вы готовитесь объяснять, вы вынуждены:

- выделять главное;
- строить порядок;
- искать причинные связи;
- подбирать пример;
- упрощать язык;
- замечать пробелы;
- думать о слушателе.

### 9.3. Протокол "объяснить за 5 минут"

```text
1. Что это такое?
2. Зачем это нужно?
3. Как это работает в простом случае?
4. Минимальный пример.
5. Типичная ошибка.
6. Чем отличается от похожего?
7. Что я сам пока не понял?
```

### 9.4. Форматы

- объяснить человеку;
- записать голосовое;
- объяснить воображаемому ученику;
- написать мини-инструкцию;
- провести 3-минутный урок;
- объяснить через пример;
- объяснить через ошибку;
- объяснить "как новичку".

### 9.5. Фишки learning by teaching

- Перед изучением поставить установку: "я буду объяснять".
- После объяснения попросить вопрос.
- Если нет слушателя, записать голосом и переслушать.
- Если объяснение распадается, вернуться к структуре.
- Использовать "пример -> антипример -> ошибка".
- Не учить другого тому, что сами не проверили.
- Не путать уверенность речи с точностью.

## 10. Generation effect / эффект генерации

### 10.1. Что это

Generation effect - материал, который человек создает сам, часто запоминается лучше, чем материал, который он просто читает.

Генерация - это:

- сформулировать ответ;
- придумать пример;
- вывести правило;
- восстановить пропущенный шаг;
- написать вопрос;
- создать антипример;
- построить схему;
- предложить гипотезу;
- сделать минимальную задачу.

### 10.2. Главное ограничение

Больше усилия не всегда лучше. Трудность должна быть продуктивной.

Продуктивная трудность:

```text
я напрягаюсь, но понимаю задачу и могу проверить результат.
```

Непродуктивная трудность:

```text
я хаотично угадываю, не понимаю критерий и не получаю feedback.
```

### 10.3. Протокол generation

После изучения:

```text
1. Закрыть источник.
2. Сформулировать главную мысль.
3. Придумать пример.
4. Придумать антипример.
5. Написать вопрос на проверку.
6. Свериться.
7. Исправить.
```

До изучения:

```text
1. Попробовать предсказать ответ.
2. Отметить неуверенность.
3. Читать источник, ища подтверждение или исправление.
```

### 10.4. Фишки generation effect

- После чтения не копировать, а переформулировать.
- Делать свои примеры, а не только чужие.
- Записывать "правило из 3 случаев".
- Придумывать контрпример к слишком широкому правилу.
- Делать маленькую задачу для себя.
- Просить ИИ проверить сгенерированный пример после своей попытки.
- Не превращать генерацию в гадание без проверки.

## 11. Pre-testing / предварительное тестирование

### 11.1. Что это

Pre-testing - попытка ответить до изучения материала.

Формула:

```text
вопрос -> попытка -> источник -> повторный ответ -> feedback
```

Цель не в правильном ответе с первого раза. Цель - усилить внимание и сделать будущий ответ значимым.

### 11.2. Когда применять

- перед чтением главы;
- перед лекцией;
- перед видео;
- перед документацией;
- перед задачей по новой теме;
- перед разбором статьи;
- перед обучающим курсом.

### 11.3. Протокол pre-testing

```text
1. Посмотреть заголовок темы.
2. Сформулировать 3-5 вопросов.
3. Попробовать ответить без источника.
4. Отметить, где угадываете.
5. Изучить материал, специально ища ответы.
6. Ответить повторно.
7. Сравнить.
```

### 11.4. Примеры вопросов

Перед статьей:

```text
О чем, вероятно, главный тезис?
Какие понятия здесь нужны?
Какая проблема решается?
Что я уже знаю?
Где я могу ошибаться?
```

Перед кодом:

```text
Какой результат я ожидаю?
Где может упасть тест?
Какой edge case вероятен?
Как бы я решил без подсказки?
```

### 11.5. Ошибки pre-testing

Ошибка: считать неправильный ответ провалом.

Исправление:

```text
неправильный ответ до обучения - ориентир внимания, если потом есть correction.
```

Ошибка: делать слишком много вопросов.

Исправление:

```text
3-5 вопросов достаточно.
```

Ошибка: устать еще до изучения.

Исправление:

```text
если вы сильно перегружены, сделать простой preview вместо жесткого pre-test.
```

## 12. Transfer practice / практика переноса

### 12.1. Что это

Transfer practice - применение идеи в новой ситуации.

Не так:

```text
повторить тот же пример теми же словами.
```

А так:

```text
взять принцип и применить к новому случаю.
```

Перенос показывает, что знание не привязано только к исходному примеру.

### 12.2. Виды переноса

#### Ближний перенос

Похожая задача с небольшим изменением.

```text
было: EmailSender
стало: SmsSender
```

#### Средний перенос

Та же идея в другом контексте.

```text
интерфейс в учебном примере -> интерфейс в тесте реального сервиса
```

#### Дальний перенос

Общий принцип переносится в другую область.

```text
feedback в программировании -> feedback в произношении -> feedback в письме
```

Дальний перенос сложнее и не гарантирован. Его нужно тренировать отдельно.

### 12.3. Протокол transfer practice

```text
1. Назвать принцип.
2. Назвать исходный пример.
3. Изменить один параметр.
4. Решить новую задачу.
5. Сравнить с исходной.
6. Записать, что осталось тем же, а что изменилось.
7. Получить feedback.
```

### 12.4. Фишки transfer practice

- Меняйте один параметр за раз.
- После успешной близкой задачи меняйте контекст сильнее.
- Всегда спрашивайте: "какой принцип переносится?".
- Делайте "пример и антипример".
- Используйте реальные задачи, а не только учебные.
- Если перенос не получается, вернитесь к self-explanation.
- Если перенос слишком легкий, добавьте interleaving.

### 12.5. Ошибки transfer practice

Ошибка: думать, что перенос появится сам после чтения.

Исправление:

```text
перенос нужно специально практиковать.
```

Ошибка: сразу пытаться делать дальний перенос.

Исправление:

```text
сначала близкий перенос, потом средний, потом дальний.
```

Ошибка: менять все условия сразу.

Исправление:

```text
менять один параметр и фиксировать эффект.
```

## 13. Сенсорная обратная связь

### 13.1. Что это

Сенсорная обратная связь - это обучение через сравнение собственной попытки с воспринимаемым образцом:

- звук;
- голос;
- произношение;
- видео;
- движение;
- жест;
- положение тела;
- визуальный результат;
- запись себя;
- ощущение в процессе действия.

Самая практичная формула:

```text
образец -> попытка -> запись себя -> сравнение -> точечная коррекция -> новая попытка
```

### 13.2. Где особенно полезно

- речь;
- произношение;
- иностранный язык;
- публичное выступление;
- пение;
- движения;
- спорт;
- письмо от руки;
- работа с инструментом;
- презентация;
- обучение через демонстрацию.

### 13.3. Протокол для произношения

```text
1. Выбрать короткий образец: слово, фраза, 1-2 предложения.
2. Послушать 2-3 раза.
3. Повторить вслух.
4. Записать себя.
5. Сравнить не всё сразу, а один параметр:
   - ударение;
   - ритм;
   - звук;
   - интонация;
   - паузы.
6. Исправить один параметр.
7. Записать повторно.
```

### 13.4. Протокол для выступления

```text
1. Сказать объяснение на 2-3 минуты.
2. Записать аудио или видео.
3. Проверить:
   - структура;
   - ясность;
   - темп;
   - лишние слова;
   - где теряется мысль.
4. Исправить только один слой.
5. Повторить.
```

### 13.5. Фишки сенсорной обратной связи

- Сравнивать один параметр за раз.
- Использовать короткие фрагменты.
- Не требовать идеальности сразу.
- Сначала копировать образец, потом варьировать.
- Записывать до и после коррекции.
- Для речи использовать медленное повторение, затем обычный темп.
- Для движений использовать видео сбоку, если это безопасно и уместно.
- Для языка слушать реальный образец, а не только читать транскрипцию.

### 13.6. Ограничения

Сенсорная обратная связь сама по себе не говорит, как исправить ошибку. Она показывает расхождение.

Нужны:

- образец;
- критерий;
- наставник или источник;
- точечная коррекция;
- повторная попытка.

Без критерия можно просто услышать, что "звучит не так", и не понять, что менять.

## 14. Как собрать технологии в одну учебную сессию

### 14.1. Новая сложная тема

```text
1. Pre-test: 3 вопроса до изучения.
2. Worked example: разобрать один пример.
3. Self-explanation: объяснить шаги.
4. Fading: решить пример с пропусками.
5. Generation: придумать свой пример.
6. Transfer: решить похожую задачу в новом контексте.
7. Feedback: проверить и записать ошибку.
```

### 14.2. Тема уже знакома

```text
1. Быстрый recall.
2. Interleaving с похожими темами.
3. Transfer practice.
4. Объяснение другому.
5. Feedback.
```

### 14.3. Навык речи или произношения

```text
1. Образец.
2. Попытка.
3. Запись себя.
4. Сравнение.
5. Одна коррекция.
6. Повтор.
7. Перенос в новую фразу.
```

### 14.4. Учебный блок на 45 минут

```text
0-5 минут:
pre-test и постановка вопроса.

5-15 минут:
worked example.

15-25 минут:
self-explanation и fading-задача.

25-35 минут:
самостоятельная задача или generation.

35-42 минуты:
transfer или interleaving.

42-45 минут:
feedback, ошибка, следующий шаг.
```

## 15. Сценарии применения

### 15.1. После видео

1. До просмотра записать 3 вопроса.
2. Во время просмотра не конспектировать всё.
3. После просмотра объяснить главную идею.
4. Привести свой пример.
5. Найти антипример.
6. Применить к своей задаче.
7. Через день проверить в другом формате.

### 15.2. После статьи

Шаблон:

```text
Главный тезис:
Какой пример у автора:
Мой пример:
Антипример:
Где применить:
С чем спутать:
Что проверить:
```

### 15.3. Для программирования

Цепочка:

```text
пример кода -> трассировка -> объяснение -> пропущенный шаг -> код с нуля -> изменение условия -> тест
```

Пример:

```text
Тема: interface в Go.

Worked example:
разобрать готовый Notifier.

Self-explanation:
почему функция принимает интерфейс?

Fading:
дописать missing fake.

Generation:
придумать свой интерфейс Logger.

Transfer:
применить к UserStore.

Interleaving:
смешать interface, pointer receiver, mock/fake.
```

### 15.4. Для языка

Цепочка:

```text
образец фразы -> повторение -> запись себя -> сравнение -> замена слов -> новая ситуация
```

Добавить interleaving:

- похожие времена;
- похожие предлоги;
- похожие звуки;
- похожие конструкции.

### 15.5. Для экзамена

Не только выучить билет, а переносить:

- ответить на прямой вопрос;
- ответить на вопрос с другой формулировкой;
- сравнить с похожей темой;
- привести пример;
- назвать ошибку;
- решить маленький кейс;
- объяснить устно без конспекта.

### 15.6. Для Obsidian

Заметка должна помогать переносу, а не только хранить текст.

Шаблон:

```text
## Суть

## Разобранный пример

## Почему шаги работают

## Мой пример

## Антипример

## Похожие темы

## Где применить

## Transfer-задача

## Ошибка и feedback
```

### 15.7. Для ИИ

ИИ полезен после собственной попытки.

Порядок:

```text
моя попытка -> вопрос к ИИ -> feedback -> повторная задача без копирования
```

Не так:

```text
дай готовое решение -> прочитал -> кажется понял
```

## 16. Фишки понимания и переноса

### 16.1. "Один пример - одна задача"

После каждого разобранного примера должна быть близкая самостоятельная задача.

### 16.2. "Сначала назвать тип"

Перед решением смешанной задачи:

```text
какой это тип и почему?
```

### 16.3. "Почему не другой метод?"

Для переноса важно не только знать правильный метод, но и понимать, почему соседний метод не подходит.

### 16.4. "Пример + антипример"

Любое сложное понятие проверять парой:

```text
где это работает;
где это похоже, но уже не оно.
```

### 16.5. "Убрать одну подсказку"

Вместо резкого перехода к самостоятельности:

```text
какую одну подсказку я уберу в следующей задаче?
```

### 16.6. "Объяснить за 3 минуты"

Если не получается объяснить за 3 минуты, знание еще не собрано.

### 16.7. "Сделать свой вариант"

После чужого примера:

```text
мой пример с тем же принципом.
```

### 16.8. "Изменить один параметр"

Для transfer practice не меняйте всё сразу. Сначала измените один элемент.

### 16.9. "Feedback должен вести к следующей попытке"

Если обратная связь не говорит, что делать дальше, она слабая.

### 16.10. "Записать себя"

Для речи и объяснения запись быстро показывает то, что не видно изнутри: темп, паузы, неясные места, произношение.

## 17. Частые ошибки

### Ошибка 1. Пассивно смотреть примеры

Исправление:

```text
пример -> прогноз -> объяснение -> задача.
```

### Ошибка 2. Убирать подсказки слишком резко

Исправление:

```text
полная поддержка -> частичная -> самостоятельная задача.
```

### Ошибка 3. Перемешивать все подряд

Исправление:

```text
смешивать похожие, но различимые типы задач.
```

### Ошибка 4. Объяснять без проверки

Исправление:

```text
объяснение -> тест / задача / источник / наставник.
```

### Ошибка 5. Генерировать без feedback

Исправление:

```text
свой пример -> сверка -> исправление.
```

### Ошибка 6. Pre-testing превращается в демотивацию

Исправление:

```text
3-5 легких вопросов, не экзамен на незнание.
```

### Ошибка 7. Перенос ждать автоматически

Исправление:

```text
специально менять контекст и задачу.
```

### Ошибка 8. Сенсорная обратная связь без критерия

Исправление:

```text
сравнивать один параметр: звук, ритм, темп, пауза, движение.
```

### Ошибка 9. AI дает решение до попытки

Исправление:

```text
сначала мой прогноз и попытка, потом AI-feedback.
```

### Ошибка 10. Слишком много методов в одной сессии

Исправление:

```text
для нового материала: пример + объяснение + близкая задача.
для знакомого: interleaving + transfer.
```

## 18. Проверка качества понимания

Сильные признаки понимания:

- могу объяснить без источника;
- могу показать простой пример;
- могу назвать антипример;
- могу выбрать метод среди похожих;
- могу применить в новой задаче;
- могу объяснить ошибку;
- могу убрать подсказку;
- могу научить другого;
- могу получить feedback и исправить действие.

Слабые признаки:

- узнаю ответ, когда вижу;
- могу повторить только формулировку автора;
- решаю только однотипные упражнения;
- не могу объяснить, почему шаг работает;
- не знаю, когда метод не применять;
- не переношу в новый контекст;
- жду готового решения от ИИ.

## 19. 7-дневный эксперимент

Цель: проверить систему понимания и переноса на одной теме.

### День 1

Тема и pre-test:

```text
3 вопроса до изучения.
1 попытка решения без подсказки.
```

### День 2

Worked example:

```text
разобрать один пример;
объяснить каждый шаг;
сделать близкую задачу.
```

### День 3

Fading:

```text
решить задачу с пропущенными шагами;
убрать одну подсказку.
```

### День 4

Generation:

```text
создать свой пример, антипример и вопрос.
```

### День 5

Transfer:

```text
применить принцип в новой задаче.
```

### День 6

Interleaving:

```text
смешать 2-3 похожих типа задач;
перед решением называть тип.
```

### День 7

Learning by teaching + review:

```text
объяснить тему за 5 минут;
записать пробелы;
выбрать следующий уровень.
```

Метрики:

- сколько самостоятельных задач завершено;
- сколько раз был feedback;
- где подсказки были нужны;
- где получилось перенести;
- где была путаница похожих тем;
- что стало понятнее;
- что удалить из системы.

## 22. Минимальная инструкция на каждый день

Если нужно максимально просто:

```text
1. Перед темой задать 3 вопроса.
2. Разобрать один хороший пример.
3. Объяснить, почему шаги работают.
4. Убрать одну подсказку и попробовать самому.
5. Сделать свой пример.
6. Применить в слегка новой задаче.
7. Смешать с похожей темой.
8. Получить feedback.
```

## 23. Главные правила

1. Понимание проверяется объяснением, примером и ошибкой.
2. Перенос сам по себе не появляется: его нужно тренировать.
3. Worked example полезен, если ведет к самостоятельной попытке.
4. Fading нужен, чтобы подсказки не стали костылем.
5. Self-explanation без feedback может закрепить ошибку.
6. Learning by teaching полезен, если объяснение перестраивает материал, а не повторяет источник.
7. Generation effect требует продуктивной, а не хаотичной трудности.
8. Pre-testing должен быть коротким и сопровождаться изучением.
9. Interleaving лучше работает с похожими, но различимыми задачами.
10. Сенсорная обратная связь требует образца, записи, критерия и повторной попытки.

Короткая формула:

```text
Не просто понял.
Объяснил.
Сделал.
Изменил контекст.
Проверил.
Исправил.
Повторил иначе.
```
