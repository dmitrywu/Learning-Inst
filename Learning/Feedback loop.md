# Feedback loop - контур обратной связи: инструкция пользователя

Режим: `ytb`

Тема: как использовать обратную связь так, чтобы она не оставалась оценкой, советом или объяснением, а превращалась в улучшение действия, повторную попытку и проверяемый прогресс.

Основные источники: локальные заметки из папок `Learning`, `Memory`, `AI` и `4sort` про feedback, учебный цикл `прогноз -> попытка -> feedback -> повтор`, deliberate practice, обучение на ошибках, active learning, self-regulated learning, competency-based education, AI-feedback и точность обучающих сигналов.

---

## 1. Короткая суть

Feedback loop (контур обратной связи) - это цикл, в котором человек:

```text
ставит цель -> делает попытку -> получает сигнал о результате -> понимает разрыв -> корректирует действие -> повторяет -> снова проверяет результат
```

Самая короткая формула:

```text
цель -> попытка -> факт -> feedback -> исправление -> повтор
```

Обратная связь становится учебной не в момент получения комментария, а в момент, когда комментарий меняет следующую попытку.

Если feedback не ведёт к действию, он остаётся информацией.

Если feedback ведёт к повторной проверке, он становится контуром обучения.

---

## 2. Главное правило

Feedback должен отвечать на три вопроса:

```text
Куда я должен прийти?
Где я сейчас относительно цели?
Что сделать дальше?
```

Если обратная связь не отвечает хотя бы на второй и третий вопрос, она часто слишком общая.

Примеры слабого feedback:

- "плохо";
- "нормально";
- "надо лучше";
- "будь внимательнее";
- "почитай ещё";
- "ты не понял";
- "хорошая работа";
- длинное объяснение без следующего шага.

Примеры сильного feedback:

- "цель - объяснить понятие с примером; сейчас есть определение, но нет условия применения; следующий шаг - добавить один контрпример";
- "ошибка не в вычислении, а в выборе метода; реши ещё одну задачу, где нужно сначала распознать тип";
- "ответ частично верный: ты правильно назвал принцип, но смешал его с похожим; перепиши различие в 3 строках";
- "тест падает не из-за функции, а из-за неверного ожидания; сначала сформулируй контракт, потом исправь проверку".

## 3. Термины

**Feedback / обратная связь** - информация о результате попытки относительно цели, критерия или ожидаемого результата.

**Feedback loop / контур обратной связи** - замкнутый цикл, где feedback приводит к корректировке и повторной попытке.

**Corrective feedback / корректирующая обратная связь** - feedback, который указывает ошибку и помогает исправить её.

**Feedforward / указание вперёд** - не только разбор прошлого результата, но и конкретный следующий шаг.

**Criterion / критерий** - условие, по которому понятно, что попытка успешна.

**Gap / разрыв** - отличие между текущей попыткой и целевым уровнем.

**Attempt / попытка** - самостоятельное действие до подсказки: ответ, задача, объяснение, черновик, решение, пример.

**Error signal / сигнал ошибки** - информация о несовпадении прогноза и факта.

**Self-regulation / саморегуляция** - способность заметить разрыв, выбрать стратегию и скорректировать обучение.

**Formative assessment / формирующее оценивание** - проверка не ради итоговой оценки, а ради улучшения следующей попытки.

**Summative assessment / итоговое оценивание** - проверка результата в конце периода; полезна для оценки, но не всегда даёт достаточно информации для обучения.

---

## 4. Что feedback loop не означает

Контур обратной связи - это не:

- просто получить комментарий;
- просто услышать оценку;
- просто попросить AI "проверь";
- просто посмотреть правильный ответ;
- просто сохранить ошибку в журнал;
- просто перечитать объяснение;
- просто получить похвалу или критику;
- просто пройти тест без разбора;
- просто накопить много замечаний.

Feedback loop есть только тогда, когда после сигнала меняется действие.

Плохой цикл:

```text
попытка -> ошибка -> объяснение -> закрыть тему
```

Сильный цикл:

```text
попытка -> ошибка -> точный feedback -> исправление -> похожая повторная задача -> новая проверка
```

---

## 5. На чём держится доказательная часть

В локальных источниках feedback описан как один из элементов доказательно разумной учебной системы.

Что поддержано лучше:

- feedback может сильно влиять на обучение, но эффект зависит от типа, уровня и качества обратной связи;
- feedback по задаче, процессу и саморегуляции обычно полезнее, чем оценочные комментарии про личность;
- ошибка учит лучше, когда есть корректирующая информация и повторная попытка;
- deliberate practice требует узкой цели, критерия, feedback и возможности повторить;
- active learning работает сильнее, когда есть действие, результат и обратная связь;
- CBE и mastery learning опираются на критерии, доказательство освоения, feedback и повтор;
- AI может ускорять подготовку feedback, но требует человеческого контроля смысла и дозировки.

Что нельзя преувеличивать:

- больше feedback не всегда лучше;
- длинный feedback может перегружать;
- feedback без попытки часто превращает ученика в пассивного получателя;
- feedback без повторной задачи не обязательно улучшает навык;
- AI-feedback может быть убедительным, но ошибочным, слишком готовым или слишком общим;
- нейробиологические исследования на мышах не доказывают напрямую учебные протоколы для человека.

Практический вывод: feedback полезен не сам по себе, а как часть цикла действия.

---

## 6. Минимальная схема feedback loop

```text
1. Цель
2. Критерий
3. Самостоятельная попытка
4. Факт
5. Разрыв
6. Следующий шаг
7. Повторная попытка
8. Проверка результата
```

Короткий шаблон:

```text
Цель:
Критерий:
Моя попытка:
Факт:
Разрыв:
Что меняю:
Повтор:
```

Если времени мало:

```text
Что хотел сделать?
Что получилось?
Что исправлю в следующей попытке?
```

## 7. Как отличить хороший feedback от плохого

Хороший feedback:

- привязан к цели;
- основан на конкретной попытке;
- указывает разрыв;
- не атакует личность;
- даёт следующий шаг;
- не перегружает количеством замечаний;
- появляется достаточно быстро;
- оставляет человеку работу;
- проверяется повторной попыткой.

Плохой feedback:

- слишком общий;
- слишком поздний;
- слишком длинный;
- оценивает личность;
- даёт готовое решение слишком рано;
- не связан с критерием;
- содержит много равнозначных замечаний;
- не показывает следующий шаг;
- не проверяется действием.

Главный тест:

```text
После этой обратной связи я понимаю, что именно сделать в следующей попытке?
```

Если нет, feedback нужно уточнить.

## 8. Виды feedback

### 8.1. Feedback по задаче

Показывает, верен ли конкретный результат.

Примеры:

- ответ правильный или нет;
- тест проходит или падает;
- решение соответствует условию или нет;
- текст отвечает на вопрос или уходит в сторону.

Полезен для базовой коррекции.

Риск: если ограничиться только "правильно/неправильно", человек может не понять, что менять.

### 8.2. Feedback по процессу

Показывает, как человек решал задачу.

Примеры:

- выбран неверный метод;
- пропущен этап проверки;
- аргумент не связан с выводом;
- слишком рано использована подсказка;
- не было самостоятельного прогноза.

Полезен для развития навыка.

### 8.3. Feedback по саморегуляции

Помогает управлять обучением.

Примеры:

- "ты слишком долго остаёшься на лёгких задачах";
- "ошибки повторяются, нужен отдельный блок на различение";
- "перед AI стоит делать собственную попытку";
- "тебе нужен критерий, иначе ты не понимаешь, завершена ли задача".

Полезен для самостоятельного обучения.

### 8.4. Feedback по личности

Примеры:

- "ты умный";
- "ты ленивый";
- "ты невнимательный";
- "у тебя нет способностей";
- "ты молодец".

Такой feedback часто слабее для обучения, потому что не показывает, что менять в действии.

Похвала может поддержать состояние, но сама по себе редко заменяет конкретную обратную связь.

## 9. Источники feedback

### 9.1. Самопроверка

Плюсы:

- доступна сразу;
- развивает саморегуляцию;
- помогает замечать пробелы.

Минусы:

- можно не увидеть собственную ошибку;
- можно переоценить понимание;
- нужен критерий.

### 9.2. Ответы и эталоны

Плюсы:

- дают быстрый факт;
- хорошо подходят для задач с проверяемым результатом.

Минусы:

- не всегда объясняют причину ошибки;
- могут спровоцировать пассивное "посмотрел решение".

### 9.3. Тесты

Плюсы:

- показывают конкретные слабые места;
- хорошо соединяются с retrieval practice;
- дают повторяемую проверку.

Минусы:

- могут проверять узнавание вместо понимания;
- не всегда показывают процесс мышления.

### 9.4. Наставник или преподаватель

Плюсы:

- может увидеть ошибку уровня процесса;
- адаптирует feedback под уровень;
- может дозировать сложность.

Минусы:

- feedback может быть слишком общим;
- наставник может решать за ученика;
- качество зависит от человека.

### 9.5. Peers / равные участники

Плюсы:

- дают альтернативный взгляд;
- полезны для объяснения и сравнения решений;
- могут снижать изоляцию.

Минусы:

- могут ошибаться;
- могут давать вкусовые комментарии вместо критериев;
- нужна культура точной обратной связи.

### 9.6. AI

Плюсы:

- быстро задаёт вопросы;
- помогает найти слабые места;
- генерирует повторные задачи;
- может адаптировать формат feedback;
- полезен для черновой проверки.

Минусы:

- может уверенно ошибаться;
- может дать слишком готовый ответ;
- может перегрузить списком замечаний;
- может заменить собственную попытку;
- требует человеческого контроля.

### 9.7. Среда и результат

Плюсы:

- результат часто объективен;
- задача показывает факт без лишних слов.

Примеры:

- текст непонятен читателю;
- задача не решается;
- тест падает;
- модель не предсказывает факт;
- выступление не укладывается во время;
- произношение не совпадает с образцом.

Минусы:

- факт не всегда объясняет причину;
- нужна интерпретация.

## 10. Feedback loop для чтения

Проблема чтения: материал кажется понятным, пока он перед глазами.

Цикл:

```text
1. Прочитать небольшой фрагмент.
2. Закрыть источник.
3. Ответить по памяти: о чём это?
4. Свериться.
5. Найти одно искажение или пробел.
6. Переформулировать своими словами.
7. Вернуться позже с вопросом.
```

Шаблон:

```text
Текст:
Мой пересказ:
Что я упустил:
Что исказил:
Новая формулировка:
Вопрос на повтор:
```

Фишка: не спрашивай "понял ли я?", спрашивай "что я могу восстановить без текста?".

---

## 11. Feedback loop для задач

Цикл:

```text
1. Выбрать тип задачи.
2. Сформулировать прогноз: какой метод подойдёт.
3. Решить без подсказки.
4. Сравнить с критерием.
5. Определить тип ошибки.
6. Получить feedback.
7. Решить похожую задачу.
8. Через интервал решить смешанную задачу.
```

Типы ошибок:

- неверно понял условие;
- выбрал не тот метод;
- правильно выбрал метод, но ошибся в шаге;
- не проверил результат;
- перепутал похожие случаи;
- слишком рано посмотрел решение.

Фишка: после feedback нужна не только "понял ошибку", а новая попытка.

---

## 12. Feedback loop для письма

Письмо хорошо подходит для feedback loop, потому что есть видимый черновик.

Цикл:

```text
1. Написать черновик.
2. Определить критерий: ясность, структура, точность, аргумент.
3. Получить feedback по 1-3 главным слабым местам.
4. Не переписывать всё сразу.
5. Исправить один слой.
6. Сравнить первую и вторую версию.
```

Виды feedback для текста:

- смысловой: что непонятно;
- структурный: где нарушена логика;
- доказательный: где не хватает основания;
- языковой: что звучит тяжело;
- редакторский: что убрать;
- читательский: где теряется внимание.

Шаблон:

```text
Цель текста:
Кому он адресован:
Критерий:
Мой черновик:
Нужен feedback только по:
Что исправил:
Что стало лучше:
```

Фишка: проси feedback не "на всё", а на конкретный слой.

---

## 13. Feedback loop для языка

Цикл:

```text
1. Сказать или написать фразу без подсказки.
2. Сравнить с образцом.
3. Найти одну ошибку.
4. Исправить.
5. Составить новую фразу с тем же элементом.
6. Повторить позже.
```

Для произношения:

```text
образец -> запись себя -> сравнение -> точечная коррекция -> повтор
```

Для грамматики:

```text
своя фраза -> correction -> правило в одну строку -> 3 новые фразы
```

Фишка: не исправляй сразу все ошибки в речи. Выбери одну, иначе feedback перегрузит внимание.

---

## 14. Feedback loop для навыков

Навык растёт через действие, критерий, feedback и повтор.

Цикл:

```text
1. Выбрать микрокомпонент навыка.
2. Выполнить короткую попытку.
3. Получить быстрый сигнал качества.
4. Исправить один элемент.
5. Повторить близкую попытку.
6. Через время проверить перенос.
```

Примеры микрокомпонентов:

- один тип задачи;
- один абзац;
- один аргумент;
- один звук;
- один жест;
- один элемент техники;
- один шаг алгоритма;
- один тип ошибки.

Фишка: deliberate practice начинается там, где feedback направлен на конкретный слабый компонент, а не на весь навык сразу.

---

## 15. Feedback loop для проектов

В проектах feedback часто приходит поздно, когда уже трудно исправлять. Поэтому нужно делать маленькие проверяемые версии.

Цикл:

```text
1. Выбрать маленький проектный шаг.
2. Задать критерий готовности.
3. Сделать черновую версию.
4. Получить feedback.
5. Исправить.
6. Проверить на пользователе, тесте, задаче или собственном критерии.
```

Примеры:

- не "сделать курс", а "проверить один урок";
- не "написать статью", а "проверить структуру аргумента";
- не "собрать проект", а "проверить один сценарий";
- не "выучить тему", а "решить одну задачу с критерием".

Фишка: чем меньше единица, тем быстрее feedback loop.

---

## 16. Feedback loop для онлайн-курса

Онлайн-курс часто даёт автоматический прогресс: проценты, галочки, завершённые уроки. Это не всегда feedback.

Настоящий feedback:

- показывает, что ты можешь сделать без урока;
- указывает пробел;
- ведёт к исправлению;
- проверяется повторной попыткой.

Шаблон после урока:

```text
Что урок обещал научить делать:
Что я сделал сам:
Где ошибся:
Какой feedback получил:
Что повторю:
```

Фишка: галочка "урок пройден" не равна замкнутому feedback loop.

## 17. Feedback loop и Anki

Anki может быть частью feedback loop, но не заменяет его.

Хороший цикл:

```text
ошибка -> вопрос -> карточка -> повтор -> новая задача
```

Плохой цикл:

```text
прочитал -> создал много карточек -> повторяю -> не применяю
```

Когда делать карточку:

- ошибка повторяется;
- понятие путается с похожим;
- факт нужен для будущих задач;
- мини-правило полезно помнить;
- вопрос проверяет извлечение, а не узнавание.

Шаблон карточки из feedback:

```text
Вопрос:
Ответ:
Типичная ошибка:
Пример:
Когда применять:
```

Фишка: карточка должна сохранять не всю обратную связь, а один полезный сигнал для будущей попытки.

## 18. Feedback loop и Obsidian

Obsidian полезен, если заметки возвращают к действию.

Шаблон заметки:

```markdown
# Feedback loop: ...

## Цель

## Критерий

## Попытка

## Факт

## Feedback

## Исправление

## Повторная попытка

## Мини-правило

## Следующий повтор
```

Неудачная заметка:

- длинный разбор;
- нет критерия;
- нет повторной попытки;
- нет следующего действия;
- нет связи с ошибками.

Фишка: если заметка не отвечает "что я сделаю иначе?", она не закрывает feedback loop.

## 19. AI-feedback: как использовать без потери мышления

AI полезен как генератор черновиков feedback, вопросов и повторных задач. Но он не должен делать первую попытку вместо ученика.

Правило:

```text
Сначала моя попытка, потом AI-feedback.
```

Хороший запрос к AI содержит:

- цель;
- критерий;
- контекст;
- собственную попытку;
- просьбу не давать готовое решение;
- ограничение по количеству замечаний;
- просьбу дать следующий шаг.

Плохой запрос:

```text
Объясни тему и скажи, как правильно.
```

Лучше:

```text
Вот моя попытка. Проверь её по критерию. Укажи 1-3 главные ошибки. Не переписывай ответ за меня. Дай один следующий вопрос и мини-задание.
```

Фишка: AI должен оставлять тебе работу. Если AI всё сделал сам, учебный контур разомкнулся.

---

## 20. Человек + AI: лучший формат

В локальных материалах есть важный вывод: ИИ полезен не как автономный оценщик, а как генератор черновиков обратной связи, которые человек проверяет, редактирует и педагогически настраивает.

Практический смысл:

- AI быстро предлагает возможные слабые места;
- человек проверяет, не ошибся ли AI;
- человек выбирает 1-2 главных пункта;
- feedback превращается в мини-задание;
- ученик делает новую попытку.

Контрольные вопросы:

```text
AI не выдумал ошибку?
Feedback соответствует моему уровню?
Он не дал готовое решение слишком рано?
Он не перегрузил меня?
Есть ли конкретный следующий шаг?
```

Фишка: AI может ускорить feedback, но не должен быть единственным судьёй качества.

---

## 21. Как просить feedback у человека

Плохой запрос:

```text
Посмотри, нормально?
```

Лучше:

```text
Мне нужен feedback по одному вопросу: понятна ли логика перехода от A к B?
```

Ещё лучше:

```text
Цель: ...
Критерий: ...
Моя попытка: ...
Где я сомневаюсь: ...
Дай, пожалуйста, 1-2 главных замечания и один следующий шаг.
```

Чем точнее запрос, тем выше шанс получить полезную обратную связь.

Фишка: не проси feedback "вообще". Проси feedback по критерию.

---

## 22. Как давать feedback другому

Формат:

```text
1. Что было целью.
2. Что получилось.
3. Где главный разрыв.
4. Почему это важно.
5. Что сделать следующим шагом.
6. Как проверить исправление.
```

Правила:

- сначала понять цель человека;
- не исправлять всё сразу;
- не атаковать личность;
- отделять факт от вкуса;
- давать ближайший шаг;
- проверять, понял ли человек feedback;
- оставлять пространство для самостоятельной доработки.

Пример:

```text
Цель была объяснить идею простым языком.
Определение есть, но пример слишком абстрактный.
Из-за этого трудно понять применение.
Следующий шаг: добавь один бытовой пример и один случай, где правило не работает.
```

Фишка: хороший feedback не демонстрирует превосходство, а улучшает следующую попытку.

---

## 23. Как принимать feedback

Получить feedback не значит автоматически принять всё.

Алгоритм:

```text
1. Отделить эмоциональную реакцию от содержания.
2. Найти конкретный сигнал.
3. Проверить, есть ли критерий.
4. Спросить уточнение, если feedback общий.
5. Выбрать один следующий шаг.
6. Сделать повторную попытку.
```

Если feedback кажется неприятным:

```text
Что здесь факт?
Что интерпретация?
Что полезно проверить?
Что я могу изменить в следующей попытке?
```

Если feedback слишком длинный:

```text
Какое одно исправление даст самый большой прирост?
```

Фишка: не нужно превращать каждый комментарий в закон. Feedback - это сигнал для проверки, а не приговор.

---

## 24. Быстрый feedback loop на 5 минут

```text
1. Что я хотел сделать?
2. Что получилось фактически?
3. Где один главный разрыв?
4. Что исправлю сейчас?
5. Как проверю исправление?
```

Применение:

- после задачи;
- после чтения;
- после объяснения;
- после ошибки;
- после урока;
- после AI-проверки;
- после разговора с наставником.

Фишка: если feedback loop занимает слишком много времени, используй только эти 5 вопросов.

---

## 25. Полный feedback loop

```text
1. Контекст
2. Цель
3. Критерии качества
4. Прогноз
5. Самостоятельная попытка
6. Факт
7. Сравнение с критерием
8. Feedback
9. Классификация ошибки
10. Мини-правило
11. Повторная попытка
12. Проверка улучшения
13. Следующий интервал
14. Недельный обзор
```

Такой формат нужен не всегда. Он полезен для сложных навыков, повторяющихся ошибок и важных проектов.

Фишка: полный формат должен применяться к важному, а не ко всему подряд.

---

## 26. Ошибка как данные

Ошибка полезна, если она отвечает на вопрос:

```text
Что изменить в следующей попытке?
```

Ошибка вредна или бесполезна, если превращается в:

- самокритику;
- избегание;
- хаотичное повторение;
- длинный архив без действий;
- доказательство "я не способен".

Шаблон:

```text
Мой прогноз:
Факт:
Расхождение:
Тип ошибки:
Мини-правило:
Повторная попытка:
```

Пример:

```text
Мой прогноз: этот метод подходит.
Факт: решение не работает на изменённом условии.
Расхождение: я не проверил ограничение.
Мини-правило: перед применением метода проверяю условие X.
Повтор: решить одну задачу, где X есть, и одну, где X нет.
```

Фишка: "я невнимательный" - плохой feedback. "Я не проверил условие X" - полезный feedback.

---

## 27. Feedback loop и психологическая безопасность

Обратная связь работает хуже, если человек воспринимает её как угрозу личности.

Это не значит, что feedback должен быть мягким и бессодержательным. Он должен быть точным и безопасным для действия.

Полезная рамка:

```text
Ошибка относится к попытке, стратегии или условию, а не к ценности человека.
```

Фразы, которые помогают:

- "в этой попытке";
- "сейчас слабое место";
- "следующий шаг";
- "проверим гипотезу";
- "исправим один элемент";
- "сравним с критерием".

Фразы, которые мешают:

- "ты всегда";
- "ты никогда";
- "у тебя нет способности";
- "это очевидно";
- "как можно было не понять".

Фишка: feedback должен снижать неопределённость, а не повышать стыд.

---

## 28. Feedback loop и уровень сложности

Feedback лучше работает, когда задача посильная.

Если слишком легко:

- feedback не нужен;
- ошибок мало;
- прогресс слабый.

Если слишком трудно:

- ошибок слишком много;
- непонятно, что исправлять;
- человек перегружается.

Оптимально:

```text
сложно, но можно сделать попытку и понять одну главную ошибку
```

Фишка: если после feedback непонятно, что повторять, задача была слишком большой или критерий слишком расплывчатый.

---

## 29. Timing: когда давать feedback

Быстрый feedback полезен, когда:

- ошибка локальная;
- нужно исправить технику;
- задача строится шаг за шагом;
- риск закрепить неверную модель высок.

Отложенный feedback полезен, когда:

- важно сначала подумать самостоятельно;
- нужно не прерывать поток;
- задача творческая;
- нужно сравнить несколько попыток.

Практическое правило:

```text
Сначала должна быть собственная попытка. Потом feedback. Потом повтор.
```

Фишка: слишком ранний feedback может украсть обучение, потому что человек не успевает столкнуться со своим прогнозом.

---

## 30. Сколько feedback давать

Частая ошибка - давать слишком много замечаний.

Для обучения обычно лучше:

```text
1-3 главных пункта + один следующий шаг
```

Чем выше перегрузка, тем меньше feedback.

Формат:

```text
Главная ошибка:
Почему важна:
Что сделать:
Как проверить:
```

Фишка: feedback должен быть дозированным. Десять замечаний часто хуже одного точного.

---

## 31. Feedback loop и недельный обзор

Саморегулируемое обучение требует не только feedback по отдельной задаче, но и обзор повторяющихся паттернов.

Вопросы недели:

```text
Какие ошибки повторялись?
Где feedback привёл к повторной попытке?
Где feedback остался только объяснением?
Какие критерии были неясными?
Где AI дал слишком готовый ответ?
Что тренировать на следующей неделе?
```

Недельный обзор не должен быть длинным. Его задача - изменить план.

Фишка: один повторяющийся тип ошибки важнее десяти случайных замечаний.

---

## 32. Feedback loop в Competency-Based Education

В CBE обучение строится не вокруг времени на тему, а вокруг доказанной компетенции.

Формула:

```text
цель -> компетенция -> критерии -> практика -> доказательство -> feedback -> повтор
```

Feedback здесь нужен, чтобы ответить:

```text
Какая компетенция уже проявилась?
Какая ещё не проявилась?
Какое доказательство нужно улучшить?
Что повторить?
```

Примеры доказательств:

- задача;
- проект;
- тест;
- объяснение;
- демонстрация;
- исправленная ошибка;
- повторная похожая задача.

Фишка: feedback без критерия компетенции часто превращается во вкусовую оценку.

---

## 33. Feedback loop в deliberate practice

Deliberate practice требует:

- узкой цели;
- концентрации;
- задачи на границе навыка;
- критерия;
- быстрой обратной связи;
- повторения после исправления.

Цикл:

```text
микрокомпетенция -> попытка -> ошибка -> feedback -> 2-3 похожие задачи -> фиксация правила
```

Если feedback есть, но нет повторных задач, это ещё не deliberate practice.

Фишка: deliberate practice - это не "позанимался подольше", а "исправил конкретный дефект".

---

## 34. Feedback loop в active learning

Active learning сильнее пассивного потребления тогда, когда есть действие и проверка.

Формула:

```text
модель -> прогноз -> задача -> ошибка -> feedback -> перенос
```

Пример:

```text
Не просто слушать объяснение.
А сформулировать модель, сделать прогноз, выполнить задание, увидеть факт, исправить модель.
```

Фишка: активность без feedback может стать суетой. Feedback без активности может стать лекцией.

---

## 35. Feedback loop и distributed practice

Feedback нужно возвращать во времени.

Сильная схема:

```text
День 0: попытка -> feedback -> исправление
День 1: recall ошибки и мини-правила
День 3: похожая задача
День 7: смешанная проверка
```

Так feedback не остаётся разовым комментарием, а превращается в долговременную настройку навыка.

Фишка: если feedback важный, у него должен быть следующий повтор.

---

## 36. Feedback loop и видимый прогресс

Прогресс лучше измерять не количеством полученных комментариев, а числом закрытых контуров.

Плохая метрика:

- получил много feedback;
- создал много заметок;
- прошёл много проверок;
- сделал длинный разбор.

Хорошая метрика:

- сделал повторную попытку;
- ошибка не повторилась;
- следующий ответ стал точнее;
- задача решена в новом условии;
- критерий стал яснее;
- слабое место стало локальнее.

Вопрос дня:

```text
Какой feedback сегодня изменил моё действие?
```

## 38. Промпты для AI

### Промпт 1. Проверить попытку без готового решения

```text
Проанализируй мою попытку как наставник.

Цель:
Критерий:
Моя попытка:

Не переписывай ответ за меня.
Дай:
1. что верно;
2. одну главную ошибку или пробел;
3. почему это важно;
4. один наводящий вопрос;
5. одно мини-задание на 5-10 минут.
```

### Промпт 2. Сжать длинный feedback

```text
Вот длинная обратная связь:
[вставить]

Сожми её до:
1. одной главной ошибки;
2. одного мини-правила;
3. одного следующего действия;
4. одной проверки результата.
```

### Промпт 3. Найти тип ошибки

```text
Помоги классифицировать ошибку.

Мой прогноз:
Моя попытка:
Факт:
Feedback:

Выбери тип ошибки:
- не понял условие;
- выбрал неверный метод;
- неверно применил метод;
- пропустил проверку;
- перепутал похожие понятия;
- другое.

Потом предложи одну повторную задачу.
```

### Промпт 4. Проверить, замкнут ли feedback loop

```text
Оцени, замкнул ли я контур обратной связи.

Цель:
Попытка:
Факт:
Feedback:
Исправление:
Повторная попытка:

Ответь:
1. контур замкнут / частично / не замкнут;
2. чего не хватает;
3. какой минимальный следующий шаг.
```

### Промпт 5. Сгенерировать повторную задачу

```text
Вот моя ошибка и feedback:

Ошибка:
Feedback:
Мини-правило:

Сгенерируй 3 задачи:
1. очень похожую;
2. с небольшим изменением условия;
3. на перенос в новую ситуацию.

Не давай решения сразу.
```

### Промпт 6. Дать feedback по тексту

```text
Проверь мой текст.

Цель текста:
Аудитория:
Критерий:
Текст:

Дай feedback только по:
1. ясности главной мысли;
2. логике структуры;
3. одному самому важному улучшению.

Не переписывай текст полностью.
```

### Промпт 7. Подготовить weekly review

```text
Вот feedback, который я получил за неделю:
[список]

Раздели:
1. повторяющиеся ошибки;
2. feedback, который привёл к действию;
3. feedback, который остался без повтора;
4. одну главную тренировку на следующую неделю.
```

## 39. Фишки

### Фишка 1. Feedback после попытки

Сначала сделай свою версию. Потом проси обратную связь.

### Фишка 2. Один разрыв за раз

Не исправляй всё. Выбери один главный разрыв.

### Фишка 3. Feedback должен иметь хвост

Хвост - это следующая попытка. Без неё feedback обрывается.

### Фишка 4. Проси вопрос, а не решение

Наводящий вопрос часто лучше готового ответа.

### Фишка 5. Записывай мини-правило

После ошибки оставь одну строку: "В следующий раз я..."

### Фишка 6. Проверяй feedback действием

Не спорь с feedback абстрактно. Сделай новую попытку и посмотри результат.

### Фишка 7. Отделяй сигнал от тона

Тон может быть неприятным, но сигнал всё равно может быть полезным. И наоборот: приятный тон не гарантирует полезности.

### Фишка 8. Дозируй AI

Проси 1-3 замечания, а не полный разбор.

### Фишка 9. Не сохраняй feedback без решения

Если сохраняешь замечание в Obsidian, добавь следующий шаг.

### Фишка 10. Повторяй через интервал

Важный feedback нужно проверить не только сразу, но и позже.

## 40. Типичные ошибки

### Ошибка 1. Просить feedback до попытки

Так человек получает объяснение вместо обучения.

Исправление:

```text
Сначала черновик, ответ, прогноз или задача. Потом feedback.
```

### Ошибка 2. Принимать оценку за feedback

"Хорошо" или "плохо" не говорит, что менять.

Исправление:

```text
Уточнить: что именно хорошо, где разрыв, какой следующий шаг?
```

### Ошибка 3. Получать слишком много замечаний

Перегрузка снижает шанс действия.

Исправление:

```text
Выбрать 1-3 главных пункта.
```

### Ошибка 4. Не делать повторную попытку

Понял ошибку, но не закрепил исправление.

Исправление:

```text
После feedback сделать похожую задачу или вторую версию.
```

### Ошибка 5. Делать feedback личностным

"Я неспособен" не помогает улучшить действие.

Исправление:

```text
Формулировать ошибку как расхождение модели и факта.
```

### Ошибка 6. Давать готовое решение слишком рано

Это снижает самостоятельную работу.

Исправление:

```text
Сначала наводящий вопрос, потом подсказка, потом решение только при необходимости.
```

### Ошибка 7. Не иметь критерия

Без критерия feedback превращается во вкус.

Исправление:

```text
Перед попыткой задать критерий качества.
```

### Ошибка 8. Хранить журнал ошибок без повторов

Архив ошибок не равен обучению.

Исправление:

```text
Каждая важная ошибка должна иметь повторную задачу или вопрос.
```

### Ошибка 9. Считать AI объективным судьёй

AI может ошибаться и звучать уверенно.

Исправление:

```text
Проверять AI-feedback по источнику, критерию или человеку.
```

### Ошибка 10. Делать feedback ritual ради ritual

Можно создать красивую систему, которая не меняет действия.

Исправление:

```text
Оставить только тот feedback, после которого есть новая попытка.
```

## 41. 7-дневный эксперимент

Цель: проверить, делает ли feedback loop обучение более управляемым.

### Гипотеза

```text
Если каждый feedback завершать повторной попыткой, то за 7 дней станет больше закрытых учебных циклов и меньше повторяющихся ошибок.
```

### Правила

```text
1. Каждый день выбрать один учебный блок.
2. Сделать самостоятельную попытку.
3. Получить feedback.
4. Выбрать один главный разрыв.
5. Сделать повторную попытку.
6. Записать мини-правило.
```

### Метрики

- была ли попытка до feedback;
- был ли feedback конкретным;
- была ли повторная попытка;
- повторилась ли ошибка;
- стал ли следующий шаг понятнее;
- не перегрузил ли feedback.

### Шаблон дня

```text
Дата:
Тема:
Попытка:
Feedback:
Главный разрыв:
Повторная попытка:
Мини-правило:
```

### Итог недели

```text
Что реально улучшилось:
Какие ошибки повторялись:
Какие источники feedback полезны:
Где feedback был лишним:
Что изменить на следующей неделе:
```

## 42. Чек-лист хорошего feedback loop

```text
Есть цель?
Есть критерий?
Была попытка до feedback?
Feedback указывает конкретный разрыв?
Он не атакует личность?
Он не перегружен?
Есть следующий шаг?
Есть повторная попытка?
Есть проверка результата?
Есть решение, когда вернуться позже?
```

Если нет повторной попытки, контур не замкнут.

## 43. Практический synthesis

Feedback loop - это не просьба "оцените меня" и не сбор замечаний. Это механизм настройки действия.

Самая полезная форма:

```text
цель -> попытка -> факт -> разрыв -> корректировка -> повтор
```

Главное:

- feedback должен появляться после самостоятельной попытки;
- он должен быть привязан к критерию;
- он должен указывать не только ошибку, но и следующий шаг;
- он должен быть достаточно коротким, чтобы по нему можно было действовать;
- он должен завершаться повторной попыткой;
- важный feedback нужно проверять позже, а не только сразу.

Если нужно оставить одно правило:

```text
Обратная связь считается полезной только тогда, когда она изменила следующую попытку.
```
